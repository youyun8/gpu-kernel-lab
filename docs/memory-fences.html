Memory Fences, In Depth

A memory fence (a.k.a. memory barrier) is an instruction that constrains the order in which memory operations become visible to other threads. This page builds the intuition from the ground up: why reordering happens, what a fence actually forbids, the four fence flavors, and how it all maps onto GPUs. Diagrams are interactive: press the buttons, drag sliders, and watch the reorderings appear and disappear.

1. Why memory ordering breaks in the first place

You wrote your program as an ordered list of statements. Almost nothing below you honors that order literally. Three separate layers are each allowed to reshuffle memory operations, as long as a single thread running alone cannot tell the difference:

LayerWhat it reordersWhy
CompilerLoads/stores across each other; hoists reads into registersOptimization: register allocation, common-subexpression elimination, scheduling
CPU core (out-of-order)Issues and completes instructions out of program orderKeep execution units busy while waiting on slow loads
Cache / store bufferDelays a store's visibility; lets later loads pass earlier storesWrites are buffered so the core need not stall on cache-coherence traffic

The key phrase is "single thread cannot tell." The guarantee that reordering preserves single-thread behavior is called sequential consistency per thread. The moment a second thread observes your memory, all those hidden reorderings become visible and your assumptions fall apart. A fence is how you claw back the ordering guarantees you need, exactly where you need them, and nowhere else (because ordering is expensive).

Mental model. Think of each thread as shouting its writes into a long hallway (the memory system). Other threads hear those shouts, but not necessarily in the order you shouted them, and not immediately. A fence is a rule like "everything I shouted before this point must be heard before anything I shout after it."

2. What a fence actually is (and is not)

A fence is a one-sided or two-sided ordering constraint placed between two memory operations in the same thread. It does three things and refuses to do a fourth:

It does: prevent specific reorderings across the fence line; force buffered writes to drain toward coherence; and (for acquire/release) create a synchronizes-with relationship that another thread can latch onto.

It does not: make an operation "instant," push data to other threads by itself, or lock anything. A fence is pure ordering, not mutual exclusion and not a broadcast. Two threads still have to cooperate: one releases, the other acquires, on the same variable.

3. The classic bug: store buffering

This is the canonical example that convinces people fences are real. Two threads, two flags, both start at 0:

// x and y both start at 0.
Thread A              Thread B
x = 1;                y = 1;
r1 = y;              r2 = x;
// Can r1 == 0 AND r2 == 0 at the end?

Intuitively "no": for both reads to see 0, both would have to run before both writes, which is impossible if the writes come first in each thread. Yet on x86, ARM, and every GPU, r1 == 0 and r2 == 0 does happen. Each core's store sits in its private store buffer while its own later load reads straight from memory, which still holds 0. The store hasn't reached the other core yet.

Interactive store-buffer simulator

Run the two threads many times. With no fence, the store buffer sometimes delays each write past the paired load, producing the "impossible" (0,0). Insert a full fence between the store and the load in each thread and that outcome disappears.

4. The four kinds of fence

Fences are classified by which pairs of operations they keep in order. Read these as "the earlier operation cannot sink below the fence, and/or the later one cannot rise above it."

FenceGuaranteesCostTypical use
LoadLoadLoads before the fence complete before loads after itcheapAfter reading a flag, before reading the data it guards
StoreStoreStores before the fence are visible before stores after itcheapAfter writing data, before writing the "ready" flag
LoadStoreLoads before the fence happen before stores after itcheapRare on its own; prevents a later write leaking past an earlier read
StoreLoadStores before the fence are visible before loads after itexpensiveThe store-buffer fix above; the only barrier a full fence really needs on x86
StoreLoad is the pricey one. It is the only ordering x86 does not give you for free, because it requires draining the store buffer. That is exactly why mfence / LOCK-prefixed atomics are the costly instructions, and why the store-buffer bug above needs a genuine full fence to fix.

5. Acquire and release: the usable abstraction

Nobody wants to hand-pick LoadLoad vs StoreStore all day. Modern languages package the two useful combinations into acquire and release semantics, which pair up like a lock:

Release (used on a store) = LoadStore + StoreStore. Everything you did before the release stays before it. Nothing sinks below. It "publishes" your prior writes.

Acquire (used on a load) = LoadLoad + LoadStore. Everything after the acquire stays after it. Nothing rises above. It "subscribes" to writes published by a release on the same variable.

// The one pattern to remember. flag is atomic.
Producer                          Consumer
data = 42;                        while (flag.load(acquire) == 0) {}
flag.store(1, release);          assert(data == 42);  // guaranteed

The release on flag guarantees data = 42 is visible before the flag flips. The acquire on flag guarantees the consumer's read of data happens after it sees the flag. Together they form a synchronizes-with edge: a promise that everything before the release is visible to everything after the matching acquire. This is the backbone of lock-free queues, spinlocks, and one-time initialization.

Rule of thumb. Release when you publish (write data, then flip a flag). Acquire when you consume (read a flag, then read the data). They must operate on the same atomic variable to synchronize. A lone release or lone acquire orders your own thread but syncs with nothing.

7. Memory fences on GPUs

This repo is a GPU kernel lab, so here is where fences bite in CUDA / HIP. GPUs have a weak memory model and an extra dimension CPUs lack: a scope hierarchy. A fence must specify not just what to order but how far the ordering reaches.

PrimitiveScopeOrders visibility to...
__threadfence_block()Thread block (CTA)Other threads in the same block
__threadfence()DeviceAll threads on the GPU (global + shared memory)
__threadfence_system()SystemOther GPUs and the host CPU (over PCIe/NVLink)
__syncthreads()BlockBarrier + fence: waits for all threads AND orders shared memory
Fence vs barrier, a common mix-up. __syncthreads() is a barrier: every thread in the block waits until all arrive, and it also fences shared memory. __threadfence() is only a fence: it orders one thread's memory operations but waits for nobody. You need the fence (not just a barrier) when threads in different blocks communicate through global memory, because __syncthreads() cannot synchronize across blocks.

The canonical GPU example: the last-block reduction

Many blocks each write a partial result to global memory, then atomically increment a counter. The last block to finish reads all partials and combines them. Without a fence, the last block may read partials that are still sitting in another block's write path, even though the counter says "everyone's done."

__global__ void reduce(float* partials, int* count, float* out) {
  partials[blockIdx.x] = blockLocalSum();   // 1. publish my partial
  __threadfence();                          // 2. make it visible device-wide BEFORE the counter bump
  if (threadIdx.x == 0) {
    int done = atomicAdd(count, 1) + 1;    // 3. announce completion
    if (done == gridDim.x) {                 // 4. I'm last: everyone's partial is visible
      float total = 0;
      for (int i = 0; i < gridDim.x; i++) total += partials[i];
      *out = total;
    }
  }
}

Remove line 2 and the code compiles, runs, and gives the right answer almost every time, which is what makes the bug so nasty. The fence is what guarantees the store on line 1 is globally visible before the atomicAdd on line 3 tells the world you finished.

8. Cheat sheet

C++ / std::atomic

// publish
data = v;
flag.store(1, memory_order_release);

// consume
while(!flag.load(memory_order_acquire));
use(data);

// full standalone fence
std::atomic_thread_fence(memory_order_seq_cst);

CUDA / HIP

__threadfence_block();   // within block
__threadfence();         // whole device
__threadfence_system();  // device + host
__syncthreads();         // barrier + shared fence

// scoped atomics (CUDA)
atomicAdd_block(p, 1);
atomicAdd_system(p, 1);
The whole page in four sentences. Hardware and compilers reorder memory to go fast, and that only shows up when a second thread looks. A fence forbids specific reorderings across a line in one thread. Acquire/release pair a publisher and a subscriber on one variable to hand data over safely. On GPUs you additionally choose a scope (block, device, system) so the ordering reaches as far as the reader lives, and remember that __syncthreads() is a barrier while __threadfence() is only a fence.
