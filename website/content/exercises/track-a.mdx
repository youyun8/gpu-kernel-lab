# Track 3 練習: GPU 基礎與效能測量

這組練習對應第 1–5 章。 Paper-and-pencil 題請先自己算, 再展開解答對照; programming 題的參考解在 <code>kernels/exercises/</code>, 可直接編譯執行。 除非題目另外標明, 假設 warp = 32、一個 memory transaction 覆蓋 128 bytes、`float` 為 4 bytes。

<Exercise id="A1" title="SIMT 與 divergence" kind="paper" level="入門">
一個 kernel 內有 `if(threadIdx.x % 2 == 0) heavy(); else light();`, 其中 `heavy()` 需 100 cycles、`light()` 需 10 cycles。 假設一個 warp 32 個 threads lockstep 執行, 忽略其他 overhead。

(a) 這個 warp 執行這段 if-else 大約需要多少 cycles? (b) 若把資料重排, 讓同一 warp 的 threads 全走同一分支, 最好情況是多少 cycles?

<Solution>
(a) warp 內同時有偶數與奇數 lane, 兩條路徑都被觸發, SIMT 必須序列化: 約 `100 + 10 = 110` cycles (兩分支各跑一遍, 期間非該分支的 lane 被 mask 掉)。

(b) 若整個 warp 都走 `heavy()`, 只需 100 cycles; 都走 `light()` 則 10 cycles。 重排資料消除 divergence, 把該 warp 的成本從 110 降到單一分支的成本。 這說明 branch divergence 的代價是「兩條路徑相加」而非「取最大」。
</Solution>
</Exercise>

<Exercise id="A2" title="Global thread index" kind="paper" level="入門">
Kernel 以 `grid = 8 blocks`、`block = 128 threads` 啟動。 請寫出 `blockIdx.x = 5`、`threadIdx.x = 20` 這個 thread 的 global index, 並說明 grid 總共覆蓋幾個元素、若要處理 `n = 1000` 個元素為何仍需邊界檢查。

<Solution>
Global index = `blockIdx.x * blockDim.x + threadIdx.x = 5 * 128 + 20 = 660`。

Grid 總共有 `8 * 128 = 1024` 個 threads。 處理 `n = 1000` 時, index 1000–1023 的 24 個 threads 超出範圍, 必須用 `if(i < n)` 擋掉, 否則越界存取造成 undefined behavior / silent corruption。
</Solution>
</Exercise>

<Exercise id="A3" title="Memory hierarchy latency 直覺" kind="paper" level="入門">
把下列存取依 latency 由小到大排序, 並各給一個數量級 (cycles) 的粗估: register、global memory (HBM)、L2 cache、shared memory/LDS。 為什麼一個把熱資料留在 register 的 kernel 通常比反覆讀 global memory 的版本快很多?

<Solution>
由快到慢: register (~1 cycle)< shared memory/LDS (~20–30 cycles)< L2 cache (~200 cycles)< global memory/HBM (~400–800 cycles)。

把重複使用的資料留在 register, 等於用一次昂貴的 global load 換來多次近乎免費的存取; 反覆打 global memory 則讓每次使用都付數百 cycles 的 latency, 且更容易變成 bandwidth-bound。 這正是 tiling 與 register blocking 的動機。
</Solution>
</Exercise>

<Exercise id="A4" title="Achieved bandwidth 計算" kind="paper" level="入門">
一個 vector add `c = a + b` 處理 `n = 2^24` 個 float, 量到的 kernel 時間是 0.75 ms。 (a) 計算實際搬移的 bytes; (b) 換算 achieved bandwidth (GB/s); (c) 若硬體 peak 是 1555 GB/s, 這是 peak 的幾 %?

<Solution>
(a) 讀 a、讀 b、寫 c 共 3 個陣列: `3 * 2^24 * 4 bytes = 3 * 16,777,216 * 4 ≈ 2.01e8 bytes ≈ 0.201 GB`。

(b) `bandwidth = 0.201 GB / 0.75e-3 s ≈ 268 GB/s` (用 `2.01e8 / 0.75e-3 / 1e9`)。

(c) `268 / 1555 ≈ 17.3%`。 Vector add 是 memory-bound, GFLOP/s 很低是正常的; 真正該看的是這個 bandwidth 百分比。
</Solution>
</Exercise>

<Exercise id="A5" title="Arithmetic intensity 與 roofline" kind="paper" level="進階">
某 kernel 做 `n` 個元素、每個元素 1 次 FMA (2 FLOP), 讀 1 個 float、寫 1 個 float。 (a) 算 arithmetic intensity (FLOP/byte); (b) 在 peak = 19500 GFLOP/s、BW = 1555 GB/s 的機器上, 它是 bandwidth-bound 還是 compute-bound? (c) 它的可達效能上限約多少 GFLOP/s?

<Solution>
(a) AI = `2 FLOP / (2 * 4 bytes) = 2 / 8 = 0.25 FLOP/byte`。

(b) ridge point = `peak / BW = 19500 / 1555 ≈ 12.5 FLOP/byte`。 0.25 遠小於 12.5, 落在 ridge point 左邊 → **bandwidth-bound**。

(c) 上限 = `BW * AI = 1555 * 0.25 ≈ 389 GFLOP/s`。 要突破這條線, 必須提高 AI (更多 reuse), 而不是優化 compute。
</Solution>
</Exercise>

<Exercise id="A6" title="計時方法的陷阱" kind="paper" level="進階">
同事用 `t0 = time(); kernel<<<...>>>(); t1 = time();` 量出 kernel 只花 5 μs, 興奮地說超快。 請指出兩個方法學錯誤, 並說明正確做法。

<Solution>
錯誤一: kernel launch 是**非同步**的, CPU 的 `time()` 只量到 launch 呼叫返回的時間, 不是 kernel 實際執行時間 (少了一個 `cudaDeviceSynchronize()` 或用 event)。

錯誤二: 沒有 **warmup** 也沒有多次 iteration, 第一次執行含 context 初始化 / JIT / cold cache, 單次量測不可靠。

正確做法: warmup 數次 → 用 `cudaEvent`/`hipEvent` 記錄 device 上的起訖 → 跑數十到數百次取平均或中位數 (見第 4 章)。
</Solution>
</Exercise>

<Exercise id="A7" title="實作 SAXPY 並量測 bandwidth" kind="code" level="入門">
以 `kernels/01-basics/vector_add.cpp` 為範本, 實作 SAXPY: `y = a * x + y`。 要求: (1) 每個 element 一個 thread、加邊界檢查; (2) 對 CPU reference 做 correctness check; (3) 用 `gklab::benchmarkKernel` 報告 achieved bandwidth。 想想每個 element 搬了幾個 float (用來算 `bytesMoved`)。

<Solution>
SAXPY 每個 element 讀 `x`、讀 `y`、寫 `y`, 共 3 次存取, 所以 `bytesMoved = 3 * n * sizeof(float)`; FLOP = `2 * n` (一次乘一次加)。 參考解在 <code>kernels/exercises/ex_a_saxpy.cpp</code>, 核心為:

```cpp
__global__ void saxpy(const float* x, float* y, float a, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) y[i] = a * x[i] + y[i];  // one FMA per element
}
```

編譯執行: `cmake --build build -j && ./build/exercises/ex_a_saxpy`。 它應和 vector add 一樣是 memory-bound, bandwidth 落在同一個量級。
</Solution>
</Exercise>

<Exercise id="A8" title="用 roofline_probe 驗證天花板" kind="code" level="進階">
編譯並執行 `kernels/02-memory/roofline_probe`。 觀察隨著每元素 FMA 次數增加 (AI 上升), GB/s 與 GFLOP/s 各如何變化? 用一句話解釋為什麼低 AI 的點 GB/s 高但 GFLOP/s 低, 高 AI 的點相反。

<Solution>
低 AI: kernel 幾乎只在搬資料, memory 系統飽和 → GB/s 接近上限, 但每 byte 只做少量 FLOP → GFLOP/s 低 (貼在 bandwidth roof)。

高 AI: 每筆資料被重複計算很多次, memory 系統閒置 → GB/s 下降, 但算術單元被餵飽 → GFLOP/s 上升 (逼近 compute roof)。

這正是 roofline 的兩段屋頂: 同一份 code, 改變 AI 就在圖上左右移動。 實測數字依硬體而異, 但趨勢一致。
</Solution>
</Exercise>

<Exercise id="A9" title="2D grid 的 thread index" kind="paper" level="入門">
一個 `1024×768` 的影像, 用 `blockDim = (16, 16)` 的 2D block。 (a) grid 該開多大 (每維幾個 block)? (b) 寫出某 thread 對應的全域 `(x, y)` 座標與線性 index (row-major)。 (c) 為什麼 kernel 裡一定要做邊界檢查 `if (x < W && y < H)`?

<Solution>
(a) `gridDim.x = ceil(1024/16) = 64`, `gridDim.y = ceil(768/16) = 48`。 (b) `x = blockIdx.x*16 + threadIdx.x`, `y = blockIdx.y*16 + threadIdx.y`, 線性 `idx = y * 1024 + x`。 (c) 當寬/高不是 blockDim 的整數倍時, 最後一排 block 會有 thread 落在影像外 (這裡剛好整除, 但通則要防護); 沒有邊界檢查會越界讀寫 → 錯誤結果或 crash。 這也是 tail effect 的來源之一。
</Solution>
</Exercise>

<Exercise id="A10" title="Warp divergence 的代價" kind="paper" level="進階">
一個 warp (32 threads) 執行 `if (tid % 2 == 0) foo(); else bar();`, `foo` 與 `bar` 各 100 條指令且無重疊。 (a) 這個 warp 大約要執行多少條指令週期? (b) 若改成 `if (tid < 16) foo(); else bar();` 代價一樣嗎? (c) 兩者對「執行時間」的影響差在哪? 給一個減少 divergence 的思路。

<Solution>
(a) SIMT 下同一 warp 的兩條分支會**序列化執行**: 先跑 foo (偶數 lane 作用、奇數關閉), 再跑 bar → 約 `100 + 100 = 200` 指令週期 (而非理想的 100)。 (b) 指令週期一樣是 200 (只要一個 warp 內同時走兩條路, 就都得跑)。 (c) 差別在**哪些 warp 受影響**: `tid<16` 讓 divergence 集中在每個 warp 內 (仍 divergent); 但若改成讓**整個 warp 走同一條路** (例如用 `tid/32` 或 block 級分工), 就沒有 warp 內分歧, 各 warp 只跑自己那條 → 無 divergence penalty。 思路: 讓分支邊界對齊 warp。
</Solution>
</Exercise>

<Exercise id="A11" title="從 copy kernel 反推 bandwidth" kind="paper" level="進階">
一個 `c[i] = a[i]` 的 copy kernel 處理 `N = 2^28` 個 `float`, 量到 kernel 時間 2.4 ms (用 `GB = 10^9 bytes`)。 (a) 這個 kernel 搬了多少 bytes (讀+寫)? (b) achieved bandwidth (GB/s)? (c) 若 GPU 理論頻寬 900 GB/s, 效率多少? 這種 kernel 適合當什麼基準?

<Solution>
(a) 讀 N + 寫 N = `2 × 2^28 × 4 bytes = 2^31 bytes ≈ 2.147e9 bytes ≈ 2.15 GB`。 (b) `2.147e9 / 2.4e-3 s ≈ 8.95e11 B/s ≈ 895 GB/s`。 (c) 效率 = `895 / 900 ≈ 99%` — 幾乎打滿。 純 copy (STREAM-like) 是量測**可達記憶體頻寬**的標準 microbenchmark, 也是所有 memory-bound kernel 的 speed-of-light 上限參考: 任何 memory-bound kernel 都不可能超過這個實測頻寬。
</Solution>
</Exercise>
