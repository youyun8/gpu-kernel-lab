# Attention Kernel: FlashAttention 與 FlashInfer

Transformer 的核心是 attention, 而 attention 的 GPU 實作是過去幾年 kernel 工程進步最集中的地方。 這一章把前面所有概念 — online softmax (第 10 章)、tiling (第 7、12 章)、warp specialization 與非同步搬運 (第 14 章) — 匯流成一個完整案例: 為什麼 naive attention 慢、FlashAttention 怎麼解、以及 serving 場景又催生了 PagedAttention 與 FlashInfer。

## Naive Attention 的問題

標準 attention 是 $O = \text{softmax}(QK^\top / \sqrt{d})\, V$。 照公式寫, 你會物化一個 $N \times N$ 的分數矩陣 $S = QK^\top$:

- **Memory 爆炸**: 序列長度 $N$, $S$ 是 $O(N^2)$。$N = 8192$ 時光一個 head 的 $S$ 就要 256 MB (fp32), 根本放不進 shared memory, 只能寫回 HBM 再讀回來做 softmax。
- **Bandwidth 主導**: attention 的算術強度不高, 把 $O(N^2)$ 的中間矩陣在 HBM 進出兩三趟, 時間全花在搬 $S$ 上 — 典型的 memory-bound (第 5 章), 而且還吃爆記憶體。

## FlashAttention: tiling + online softmax

FlashAttention 的洞見: **永遠不要物化整個 $S$**。 把 $Q, K, V$ 沿序列切成 block, 用兩層迴圈逐塊計算, softmax 用第 10 章的 online 公式邊算邊 rescale:

```text
for each block of Q (outer):
    m = -inf, l = 0, acc = 0          # running max, denom, output
    for each block of K, V (inner):
        S_ij = Q_i @ K_j^T            # small tile, lives in shared memory / registers
        m_new = max(m, rowmax(S_ij))
        P_ij  = exp(S_ij - m_new)     # unnormalized probabilities for this tile
        l = l * exp(m - m_new) + rowsum(P_ij)   # rescale running denom (ch. 10)
        acc = acc * exp(m - m_new) + P_ij @ V_j # rescale running output
        m = m_new
    O_i = acc / l                     # finalize
```

關鍵收益: 中間的 $S_{ij}$ 只有 block 大小、全程留在 shared memory/register, 從不寫 HBM。 HBM traffic 從 $O (N^2)$ 降到 $O (N)$ — 這是把「IO-awareness」帶進 kernel 設計的教科書範例, 也讓 attention 從 memory-bound 翻身接近 compute-bound。 Backward 用同樣的 tiling, 靠重算 (recompute) $S$ 避免存下整個矩陣。

虛擬碼落到真的 kernel 裡, online 更新的核心就這幾行 (取自 lab 的 `attention_naive`, 一個 block 處理一個 query row):

```cpp
float m_old = m_shared;
if (e == 0) {
  float s = score_parts[0] * scale;      // this key's score
  float m_new = fmaxf(m_old, s);
  p_shared = expf(s - m_new);
  d_shared = d_shared * expf(m_old - m_new) + p_shared;  // rescale old denom
  m_shared = m_new;
}
__syncthreads();

// Every thread rescales its output accumulator the same way, then adds p * v.
acc = acc * expf(m_old - m_shared) + p_shared * v[key * kHeadDim + e];
```

整個 $N \times N$ 的分數矩陣自始至終只存在於這幾個 running 變數裡。

<Callout variant="insight" title="這章其實沒有新東西">
FlashAttention 不引入任何你沒學過的原語: 它就是 online softmax (第 10 章) + tiling (第 7、12 章) + 對 memory 階層的執著 (第 3、5 章)。 這正是本站的論點 — 頂尖 kernel 是基礎技巧的深度組合, 不是魔法。
</Callout>

## 版本演進: 榨乾每一代硬體

- **FlashAttention-1** (2022): 確立 tiling + online softmax + recompute backward。
- **FlashAttention-2** (2023): 重排迴圈減少非 matmul 的 FLOP、更好的 warp 分工、把 work 沿序列維度平行化以填滿 GPU (第 13 章的 occupancy/quantization 考量)。
- **FlashAttention-3** (2024): 針對 Hopper — 用 TMA 搬 tile、`wgmma` 做 MMA、warp specialization 讓 softmax 與 matmul 重疊 (全是第 14 章的機制), 並支援 fp8。

每一代的進步都對應「更充分利用新硬體特性」, 而非改演算法本身 — 讀它們的 changelog 幾乎就是第 12–14 章的實戰註解。

## Serving 場景: PagedAttention 與 FlashInfer

訓練時序列長度整齊; LLM **推論**時每個 request 的 KV cache 長度不一、還在動態增長, 這帶來新問題:

- **PagedAttention** (vLLM): 借作業系統分頁的想法, 把 KV cache 切成固定大小的 block 分散存放, 用一張 block table 索引。 消除了「為最長序列預留連續記憶體」的浪費, 讓 batch 內不同長度的 request 高效共存 — 這是 kernel 設計配合記憶體管理的典範 (attention kernel 必須能從非連續的 KV block 讀取)。
- **FlashInfer**: 專為 LLM serving 的 attention kernel library, 針對 decode (每步一個 token, 極度 memory-bound)、prefill、以及各種 KV layout 與 mask 提供高度優化且可組合的 kernel; 被 vLLM、SGLang、TensorRT-LLM 採用。 它把「serving 場景的 attention」當成一個獨立的優化問題, 而不只是訓練 kernel 的縮小版。

Decode 階段特別值得注意: batch=1、query 只有一個 token, 這是純 memory-bound (讀整個 KV cache 只算一步), 優化重點從 compute 轉向「KV cache 的搬運與量化」— 與訓練/prefill 的 compute-bound 大 tile 是完全不同的 regime。

## 你會在哪裡遇到這些

幾乎無所不在, 而且多半不用自己寫: PyTorch 的 `F.scaled_dot_product_attention` 會自動選 FlashAttention/memory-efficient backend; HuggingFace 用 `attn_implementation="flash_attention_2"` 開啟; vLLM/SGLang 依場景在 FlashAttention、FlashInfer、PagedAttention、AITER (AMD, 第 23 章) 之間切換。 你的價值在於: 讀得懂 profiler 說 attention 卡在哪、知道該換哪個 backend、以及理解為什麼。

## Attention Variant Map

不同 attention 變體對 kernel 的要求不同:

- **Causal self-attention**: mask 是三角形, tile 排程可跳過未來區塊。
- **Cross-attention**: Q 長度和 KV 長度不同, prefill/decode 的 reuse pattern 改變。
- **Grouped-query / multi-query attention**: 多個 query heads 共用 KV heads, decode 讀 KV 的 bandwidth 降低。
- **Sliding-window attention**: 每個 token 只看局部窗口, tile scheduler 可限制 K/V block 範圍。
- **Paged KV cache**: KV 不連續, kernel 需要讀 block table, coalescing 和 TLB/cache locality 變成核心。
- **Quantized KV**: fp8/int8 KV cache 降 bytes, 但每次讀取要 dequant scale, scale layout 會影響 decode latency。

因此 attention backend 不是單選題。 Prefill、decode、長上下文、短上下文、量化、不同 mask 可能各自對應不同 kernel。 Production 引擎通常維護一個 dispatch table, 而不是「FlashAttention 永遠最快」。
