# GPU 是什麼樣的機器?

同樣是矽晶片, 為什麼 CPU 只有幾十個核心, GPU 卻有上萬個? 如果 GPU 這麼多核心, 為什麼跑一段普通的 if-else 程式反而比 CPU 慢? 這一章先把 GPU 的設計哲學講清楚 — 後面每一個優化技巧, 都是這個哲學的推論 — 並用兩個小實驗 (device query、branch divergence) 讓你親手驗證。

<Callout variant="insight" title="先給直覺">
CPU 是短跑選手: 少數幾條非常快的核心, 配上巨大的 cache 與複雜的 branch predictor, 目標是把「單一任務」跑到最低 latency。 GPU 是搬運大隊: 成千上萬條相對慢、但極多的執行單元, 目標是把「大量相同任務」的 throughput 拉到最高。 你不會叫一個短跑選手去搬一萬箱貨。
</Callout>

## Throughput vs. Latency

CPU 花大量電晶體在「讓一條指令流盡快完成」: 亂序執行、分支預測、多層 cache。 GPU 反過來, 把電晶體幾乎都投在 ALU 上, 靠「同時有海量的工作在排隊」來掩蓋每一項工作的延遲。 當一個 warp 在等 memory, scheduler 立刻切到另一個 ready 的 warp, 執行單元幾乎不閒著。 這叫 latency hiding, 是理解 GPU 效能的第一把鑰匙。

這也解釋了為什麼「一段有很多分支、彼此相依的序列程式」在 GPU 上不快: 它既沒有足夠的平行度餵飽執行單元, 又觸發 SIMT 的 divergence 懲罰。

用一個粗略的數字建立體感: 一次 global memory 存取約 400–800 cycles。 CPU 用 cache 與預取把這個延遲「藏在單一執行緒後面」; GPU 則需要同時有幾十個 warp 排隊, 讓 scheduler 永遠找得到「下一個可以發射指令的 warp」。 所以 GPU 程式的第一要務是: 給它足夠多、足夠獨立的工作。

## SIMT 執行模型

GPU 以 warp (NVIDIA, 32 threads) 或 wavefront (AMD, 64 threads, RDNA 可 32) 為單位排程: 同一個 warp 裡的所有 threads 在同一個 cycle 執行同一條指令, 只是各自的 register 資料不同。 這就是 SIMT (Single Instruction, Multiple Threads)。

推論很直接: 如果 warp 裡的 threads 走進不同的 if 分支 (branch divergence), 硬體必須把兩條路徑輪流跑一遍, 效能折半。 後面章節談的 coalescing、bank conflict、warp shuffle, 全都圍繞「一個 warp 作為一個整體」這個事實。

這件事可以直接量出來。 下面兩個 kernel 做的總工作量完全相同, 差別只在分支條件跟不跟 warp 邊界對齊:

```cpp
// Uniform: the condition depends only on the warp index, so all 32
// lanes of a warp take the same path. No serialization.
__global__ void branchUniform(const float* in, float* out, int n) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i >= n) return;
  int warp = i / 32;
  out[i] = (warp % 2 == 0) ? heavyPath(in[i]) : lightPath(in[i]);
}

// DIVERGENT on purpose: even and odd lanes of the SAME warp branch
// apart, so every warp executes heavyPath AND lightPath back to back.
__global__ void branchDivergent(const float* in, float* out, int n) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i >= n) return;
  out[i] = (i % 2 == 0) ? heavyPath(in[i]) : lightPath(in[i]);
}
```

Lab 的 `branch_divergence` 會實測兩者: divergent 版明顯更慢, 慢多少取決於兩條路徑的成本差 — 這就是「warp 是執行的原子單位」的直接代價。

SIMT 和傳統 SIMD (CPU 的向量指令) 的差別在於程式模型: SIMD 要你手動打包向量、處理邊界; SIMT 讓你用「每個 thread 一份 scalar 程式」的方式思考, 由硬體負責把 32/64 個 threads 綁成一組執行。 寫起來像 scalar code, 跑起來是向量機 — 這是 CUDA 成功的關鍵設計。

## SM / CU: GPU 的基本組織單位

整張 GPU 由數十到上百個 SM (Streaming Multiprocessor, NVIDIA) 或 CU (Compute Unit, AMD) 組成。 每個 SM/CU 內含:

- 數個 warp scheduler 與發射埠 (每 cycle 挑幾個 ready warp 發射指令)。
- 一批 ALU / FMA 單元、特殊功能單元 (SFU, 算 `exp`、`sin` 等)、以及矩陣加速單元 (Tensor Core / Matrix Core)。
- 一塊 register file (所有駐留 thread 均分) 與一塊 shared memory / LDS。

一個 thread block (CTA) 會被整個放到某一個 SM/CU 上執行, 直到跑完都不搬家; 一個 SM/CU 可以同時駐留多個 block。 這就是為什麼 block 內的 threads 能便宜地同步與共享 shared memory, 而跨 block 的協作必須經過 global memory — 硬體上它們可能在晶片的兩端。

<Callout variant="info" title="數量級體感">
A100 有 108 個 SM, 每個 SM 可駐留 64 個 warp = 2048 threads; MI300X 有 304 個 CU。 也就是說, 要「填滿」一張現代 GPU, 你至少需要數十萬個 threads 在飛。 第 13 章的 wave quantization 與 tail effect, 就是「工作量不夠填滿機器」時的具體病徵。
</Callout>

你的機器有幾個 SM/CU? 用 runtime API 問它:

```cpp
int device_count = 0;
GPU_CHECK(gpuGetDeviceCount(&device_count));
for (int dev = 0; dev < device_count; ++dev) {
  GPU_CHECK(gpuSetDevice(dev));
  int sm_count = 0;
  GPU_CHECK(gpuGetMultiprocessorCount(&sm_count));  // cudaDeviceGetAttribute / hipDeviceGetAttribute
  std::printf("device %d: %d SM/CU, ~%d threads to fill at 2048 resident/SM\n",
              dev, sm_count, sm_count * 2048);
}
```

Lab 的 `device_query` 就是這段程式。 在任何新機器上先跑它一次, 把「這張卡要多少 threads 才餵得飽」變成一個具體數字。

## CUDA 與 ROCm 生態系對照

本站兩個平台並行。 名詞會不斷交錯出現, 先建立對照表:

| 概念 | NVIDIA (CUDA) | AMD (ROCm/HIP) |
| --- | --- | --- |
| 執行單位 | warp (32 threads) | wavefront (64 threads; RDNA 可 32) |
| 運算核心叢集 | SM (Streaming Multiprocessor) | CU (Compute Unit) |
| thread block | block / CTA | block / workgroup |
| on-chip 快取記憶體 | shared memory | LDS (Local Data Share) |
| 低階 ISA | PTX → SASS | GCN / CDNA / RDNA ISA |
| 編譯器 | nvcc | hipcc |
| 矩陣加速單元 | Tensor Core (WMMA/mma) | Matrix Core (MFMA) |
| BLAS library | cuBLAS / cuBLASLt | hipBLAS / hipBLASLt |
| profiler | Nsight Systems / Nsight Compute | rocprofv3 / omniperf (rocprof-compute) |
| kernel template library | CUTLASS | Composable Kernel (CK) |

<Callout variant="warn" title="可攜性陷阱先劇透">
Warp = 32 vs. wavefront = 64 是最常見的可攜性地雷。 任何寫死 `32` 的 warp-level 程式碼 (shuffle、mask、reduction 步數) 搬到 AMD 都可能出錯。 第 9 章會專門處理這件事; 現在先記住這個差異。
</Callout>

HIP 的 API 幾乎是 CUDA runtime 的鏡像 (`cudaMalloc` ↔ `hipMalloc`), 多數 CUDA 程式可用 `hipify` 半自動轉換。 本站的 kernels 盡量用一層 portability header 讓同一份程式碼在兩個平台編譯。

## 什麼樣的問題適合 GPU

把設計哲學倒過來讀, 就是適用性判斷:

- **大量、同質、彼此獨立的工作** (elementwise、矩陣乘、影像處理、attention): GPU 的主場。
- **資料夠大**: 資料太小時, kernel launch 與 PCIe 傳輸的固定成本會吃掉所有收益 (第 17 章)。
- **分支少、存取規律**: 高度 data-dependent 的指標追逐、狀態機、遞迴解析, 留在 CPU 上更快。

實務上的系統都是混合體: CPU 負責控制流與資料準備, GPU 負責數值密集的主體。 優化的起點永遠是先確認「這段工作真的該在 GPU 上嗎」。

## Optimization Techniques Preview

後面所有技巧都可以回到本章的硬體模型:

- **Expose parallelism**: 用更多 independent elements、tiles、rows、heads 或 requests 餵飽 SM/CU。 資料太小時, batch、fuse 或用 persistent kernel 攤平 launch overhead。
- **Make work uniform**: 讓同一 warp/wavefront 內的 lanes 做相似工作。 遇到 data-dependent branch 時, 先排序、分桶或拆成多個 kernel, 不要讓一個 warp 裡一半 lanes 空等。
- **Move less data**: GPU 算術很強, HBM/PCIe 才常是瓶頸。 Coalescing、tiling、fusion、quantization、prepack 都是在減少昂貴資料搬移。
- **Reuse on chip**: 被多次使用的資料應該盡量留在 register、shared memory/LDS 或 L2。 GEMM、stencil、attention 都是把 reuse 顯式化的問題。
- **Pick the right engine**: scalar CUDA core、Tensor Core/MFMA、copy engine、NCCL/RCCL communication engine ([第 33 章](/chapters/g33-nccl-rccl-collectives)) 各有用途。 高效系統會讓它們重疊, 而不是只盯著一個 kernel。

一個實務例子: 如果你要把 100 個小 tensor 各做一次 bias + activation, 直接 launch 100 個小 kernel 通常不划算。 更好的路線可能是先把 tensor batch 起來、用一個 fused kernel 處理, 或讓 framework 用 CUDA Graph/HIP Graph replay 固定 DAG。 這不是語法問題, 而是「GPU 喜歡大量同質工作」這個設計哲學的直接推論。

預期結果與判讀:

- CMake log 會印 `gpu-kernel-lab platform: CUDA` 或 `HIP`; 兩者皆無時印 `NONE` 並警告只能做 syntax-only 檢查。
- `device_query` 印出每張卡的 SM/CU 數。 例如 A100 是 108 個 SM → 約 221,184 threads 才能填滿; 這個數字就是「你的問題夠不夠大」的門檻。
