# CTA Scheduling: Swizzle 與 Persistent Kernel

第 5 章處理「tile 數**不夠**」的情況。 這一章相反: 當 tile 數**夠多**時, 它們被排上 SM 的**順序**會大幅影響 L2 cache 的命中率與尾端行為。 同一組 tile、同樣的 compute, 只是換個排程順序, 就能差 10–30%。

<Callout variant="insight" title="落到真實場景: 只改排程就多 20% 的那次 commit">
「換排程差 10–30%」聽起來抽象, 但在 CUTLASS/Triton 的 issue 與 changelog 裡這是實打實的 commit: 對一個大方陣 GEMM, 把 threadblock 的 rasterization 從 row-major 改成 grouped swizzle, L2 hit rate 從約 50% 拉到約 75%, end-to-end 快 20% — 沒動任何一行 compute。 對正在調 LLM 訓練 GEMM 的人, 這是「幾乎免費」的一檔: 先試 swizzle 與 persistent 這類排程旋鈕, 再去碰更難的 mainloop。 本章就是這兩個旋鈕。
</Callout>

## CTA Swizzle — 換 launch 順序換 L2 命中

GEMM 的每個 output tile 都要讀一整條 A 的 row-band 與一整條 B 的 column-band。 相鄰的 output tile 共享這些 band。 問題是: **哪些 tile 同時在跑**, 決定了它們共享的 band 能不能留在 L2 被重複命中。

預設的 row-major launch 順序讓同時活躍的 CTA 攤在 C 的一整列上 — 它們各自需要不同的 B column, 共享的 A row 很快被擠出 L2。 **Swizzle (rasterization)** 把 launch 順序重排成一塊塊, 讓鄰近時間啟動的 tile 在空間上也相鄰:

<CtaSwizzleFigure />

```cpp
// Map the linear blockIdx to a swizzled (tileRow, tileCol) so consecutively
// scheduled CTAs stay in a compact 2D block -> shared A/B bands hit in L2.
int tilesN   = (n + BN - 1) / BN;
int groupSz  = kSwizzle * tilesN;          // kSwizzle rows form one group
int group    = blockIdx.x / groupSz;
int inGroup  = blockIdx.x % groupSz;
int tileRow  = group * kSwizzle + inGroup % kSwizzle;
int tileCol  = inGroup / kSwizzle;
```

這個 `kSwizzle` (一組幾列) 是個純排程旋鈕, 不改變任何計算結果, 卻能顯著改善 L2 reuse。 它也是第 7 章 autotuning 會一起搜的參數之一。

<Callout variant="insight" title="Swizzle 是免費的午餐嗎">
幾乎是: 它不增加 FLOP、不增加 memory traffic, 只重排順序。 但最佳的 swizzle 因數依賴 L2 大小、tile 尺寸與矩陣形狀, 沒有一體適用的值 — 所以 library 用 heuristic 或 autotune 來挑。
</Callout>

## Persistent Kernel — 常駐 CTA 自己領工作

傳統 GEMM 一個 output tile 一個 CTA, tile 多時就 launch 成千上萬個 CTA, 靠硬體排程器一波波換上換下。 每次 CTA 上下場都有 overhead, 而且 tile 之間無法共享已載入 register/shared 的資料。

**Persistent kernel** 反過來: 只 launch 剛好填滿 GPU 的 CTA 數 (= SM 數 × 每 SM CTA 數), 讓它們**常駐**, 每個 CTA 跑一個迴圈, 從一個 **work queue** 反覆領取下一個 tile:

```cpp
// Persistent GEMM: launch exactly gridSize = numSMs * ctasPerSM CTAs, then
// each one loops pulling tile indices from a shared atomic counter until drained.
__global__ void sgemmPersistent(/* ... */, int numTiles, int* nextTile) {
    for (int tile = atomicAdd(nextTile, 1); tile < numTiles; tile = atomicAdd(nextTile, 1)) {
        int tileRow = tile / tilesN, tileCol = tile % tilesN;
        computeOneTile(tileRow, tileCol);   // full tiled/MMA mainloop for this tile
    }
}
```

好處:

- **省下反覆的 CTA launch / teardown overhead**, 尤其對很多小 tile 的情境。
- **動態負載平衡**: 快的 CTA 自然領到更多 tile, 天然抵抗 tail effect (和 Stream-K 精神相通, Stream-K 常就是 persistent kernel 的一種)。
- **跨 tile 保留狀態**: 常駐 CTA 可以攤銷 prologue、甚至在 tile 間 pipeline。

代價是要自己管 work queue 的 atomic counter, 且 occupancy 要抓準 (launch 太多會回到被換上換下的老問題, 太少則填不滿)。 這是第 9 章 track「persistent kernels 與 work queues」的核心, 也是現代 library GEMM 的預設結構。

<Callout variant="insight" title="落到真實場景: Hopper 上的 persistent warp-specialized GEMM">
persistent kernel 不是學術玩具 — 它就是 CUTLASS 3.x 在 Hopper 上 GEMM 的預設骨架: launch 剛好塞滿 GPU 的 CTA, 讓一部分 warp 專門用 TMA 搬資料 (producer)、另一部分專門跑 MMA (consumer), tile 之間從 work queue 連續領, prologue 只付一次。 對「同一個 kernel 要連算幾百個中小 GEMM」的 MoE 或 batched 場景, persistent 省下的反覆 launch/teardown overhead 特別明顯。 代價是那個 atomic `nextTile` counter 在 tile 很小時會變成 contention 熱點, 所以 library 常改用每個 CTA 自己算 index 的無 atomic 方案。
</Callout>

## 把第 5、6 章接起來

| 問題 | 症狀 | 工具 |
| --- | --- | --- |
| tile 數 < 硬體並行度 | 大半 SM 空轉 | Split-K / Stream-K (第 5 章) |
| tile 數 ≫ 並行度, 但 L2 沒吃好 | memory 反覆重載同樣的 band | CTA swizzle (本章) |
| 很多小 tile, launch overhead 高 / tail | 排程開銷、尾端不平衡 | persistent kernel (本章) |

這三者可以疊加: 一個 production GEMM 常常是「persistent kernel + swizzle 排程 + 在 skinny shape 時切到 Stream-K」的組合 — 而選哪個組合、參數設多少, 就交給第 7 章的 autotuning。

<Callout variant="insight" title="落到真實場景: 拆一個 production GEMM 的組合拳">
把這張表對到一次真實的 profiler session: 你在跑一個 GPT 訓練 step, 發現某個 attention output projection 的 GEMM 慢。 先看 tile 數 vs SM 數 — 夠多, 排除 Split-K; 再看 L2 hit rate 偏低 — 開 swizzle; 又看到很多小 tile 讓 launch 佔比高 — 換 persistent。 三個症狀對三個工具, 疊起來就是這一格。 真正的 library (cuBLASLt/CUTLASS) 把這整套組合連同參數丟給第 7 章的 autotuner 去搜, 但你得先看得懂每格在解什麼, 才知道它搜出來的組合合不合理。
</Callout>
