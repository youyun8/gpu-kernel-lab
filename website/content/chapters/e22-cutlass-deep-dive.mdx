# CUTLASS 深入: 分層設計與 CuTe

第 15 章把 CUTLASS 放上了生態地圖; 這一章打開引擎蓋。 CUTLASS 是 NVIDIA 開源的 GEMM/attention kernel template library — cuBLAS 的許多 kernel、PyTorch 的 fp8 GEMM、FlashAttention-3、vLLM 的量化 GEMM 都建立在它之上。 讀懂它的分層, 你就讀懂了現代 NVIDIA GPU 上「工業級 kernel」的標準結構。

## 分層: 從 device 到指令

CUTLASS 把一個 GEMM 拆成嵌套的層級, 每層對應第 12 章優化階梯的一站:

| 層級 | 職責 | 對應本站章節 |
| --- | --- | --- |
| Device | kernel launch、grid 規劃、Split-K/Stream-K scheduler | 第 13 章 |
| Kernel | CTA 的 persistent loop、prologue/epilogue 串接 | 第 13、14 章 |
| Threadblock (CTA) | global → shared 的 tile 搬運、multi-stage pipeline | 第 12 章 Step 1/4 |
| Warp / Warpgroup | shared → register、餵 MMA 指令 | 第 12 章 Step 2/5 |
| Instruction | `mma` / `wgmma` 指令的封裝 | 第 12 章 Step 5 |
| Epilogue | accumulator → bias/activation/dtype 轉換 → global | 第 12 章 epilogue |

每一層都是可替換的 template 參數: 換 tile 尺寸、換 scheduler (Stream-K)、換 epilogue (bias + GELU)、換 dtype (fp8/int4), 底下其他層的優化原封不動 — 這就是「可組合」的意思, 也是它相對黑盒 cuBLAS 的根本差異。

## CuTe: 3.x 的地基

CUTLASS 3.x 重寫的核心是 **CuTe** — 一套描述「layout」的代數。`Layout = Shape : Stride`, 把「這塊資料在記憶體/register 裡怎麼擺」變成可組合、可運算的一等公民:

```cpp
// CuTe: a layout maps logical coordinates to memory offsets.
// A 8x16 row-major tile:
auto layout = make_layout(make_shape(Int<8>{}, Int<16>{}),
                          make_stride(Int<16>{}, Int<1>{}));
// Compose with a swizzle to avoid bank conflicts (ch. 7) while keeping the
// logical view unchanged:
auto swizzled = composition(Swizzle<3, 3, 3>{}, layout);
// Partition the tile across a thread layout; each thread sees its slice:
auto frag = local_partition(tensor, thr_layout, threadIdx.x);
```

第 7 章的 padding/XOR swizzle、第 12 章 MMA fragment 的怪異佈局, 在 CuTe 裡都是 layout 代數的一次 `composition`。 這解決了手寫 MMA kernel 最痛的部分: 佈局的正確性由型別系統保證, 而不是靠人肉推 index。

## 實際用起來像什麼

多數人不從零組裝, 而是實例化現成的 collective:

```cpp
// CUTLASS 3.x: declare a Hopper GEMM by composing policies, then run it.
using Gemm = cutlass::gemm::device::GemmUniversalAdapter<
    cutlass::gemm::kernel::GemmUniversal<
        Shape<int, int, int>,                       // problem shape (M, N, K)
        CollectiveMainloop,                         // TMA + wgmma pipeline
        CollectiveEpilogue,                         // e.g., LinCombPerRowBias + GELU
        cutlass::gemm::StreamKScheduler>>;          // tile scheduler (ch. 13)

Gemm gemm;
gemm.initialize(args, workspace);
gemm.run(stream);
```

選 config 不用猜: **CUTLASS Profiler** (`cutlass_profiler --operation=gemm --m=4096 ...`) 會枚舉所有合法變體並實測 — 和第 13 章 cuBLASLt 的離線 tuning 是同一件事, 只是候選集是你自己編譯出來的。 Python 使用者另有 `cutlass` Python package 與 DSL, 能在 notebook 裡 JIT 實例化 GEMM。

## 誰在用它

- **PyTorch**: `torch._scaled_mm` (fp8 GEMM)、部分 quantized kernel、`scaled_dot_product_attention` 的 memory-efficient backend (源自 xFormers 的 CUTLASS attention)。
- **FlashAttention-3**: 直接以 CUTLASS/CuTe 寫成, 使用 TMA + wgmma + warp specialization (第 14 章)。
- **vLLM / TensorRT-LLM**: 量化 GEMM (fp8/int8/int4)、MoE grouped GEMM 等大量取自 CUTLASS 或其變體 (第 25 章)。
- **DeepSeek DeepGEMM 等第三方**: 以 CuTe 為基礎的客製 GEMM 也越來越常見。

<Callout variant="insight" title="學習路徑建議">
不要從 CUTLASS 原始碼的最深處開始啃。 有效的順序: (1) 用 cutlass_profiler 感受 config 空間有多大; (2) 跑 examples/ 下的基本 GEMM, 改 tile 尺寸與 epilogue 觀察效能變化; (3) 讀 CuTe 的 layout 教學, 直到你能解釋第 7 章的 swizzle 如何被表達; (4) 這時再讀 mainloop 原始碼, 你會發現它就是第 12–14 章的組合。
</Callout>

## Customize Without Forking

CUTLASS 的正確使用方式通常不是 fork 後手改 mainloop, 而是替換 extension points:

- **Epilogue visitor**: bias、activation、scale、clamp、quantize 放在 epilogue tree。
- **Scheduler**: standard、Split-K、Stream-K、persistent scheduler 按 shape 選。
- **Tile shape**: 用 profiler 找候選, 再把少數最佳 config 編進 dispatch table。
- **Layout**: 用 CuTe composition 表達 swizzle/prepack, 避免手寫 offset。
- **Collective mainloop**: 只有當現有 pipeline 不支援你的資料流時才自訂。
- **Manifest/dispatch**: production 應把 supported dtype/layout/arch 明確列出, fallback 到 cuBLASLt。

這樣做能保留 upstream 更新的好處。 手改深層 pipeline 只有在你明確知道現有 collective 缺哪個硬體特性時才值得。
