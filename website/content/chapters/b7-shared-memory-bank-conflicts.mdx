# Shared Memory 與 Bank Conflicts

你把資料搬進 shared memory, 期待它像廣告說的那樣快, 結果 profiler 顯示 shared memory 的存取效率只有 1/32。 問題不在 shared memory 慢, 而在你不小心讓一個 warp 的 threads 全擠向同一個 bank。 這一章用經典的 matrix transpose 三部曲, 把 bank conflict 講透。

## 為什麼需要 Shared Memory: Tiling 的動機

Shared memory 是 block 內可控的高速暫存。 當多個 threads 會重複讀到同一批 global 資料時 (transpose、GEMM、stencil、reduction), 先協力載入 shared memory, 之後從 shared memory 反覆讀, 就能把昂貴的 global load 攤薄。 這就是 tiling。

## Bank 結構

Shared memory 被切成多個 bank, 能同時服務不同 bank 的存取。 NVIDIA 是 32 個 bank (對應 warp 32 lanes), AMD LDS 也是分 bank 的結構 (CDNA 為 32 banks 配 64-lane wavefront)。 規則:

- 一個 warp 的 threads 存取 **不同 bank** → 全平行, 一個 cycle 完成。
- 多個 threads 存取 **同一 bank 的不同 address** → bank conflict, 序列化。
- 所有 threads 讀 **同一 address** → broadcast, 沒有 conflict。

Bank 由 address 決定: `bank = (address / 4 bytes) % 32`。 所以「stride 剛好是 32 的倍數」的存取模式, 會讓整個 warp 撞進同一個 bank, 變成 32-way conflict。

N-way conflict 的代價就是 N 倍時間: 硬體把這條存取指令重播 (replay) N 次, 每次服務一個 bank 內的一個 address。 profiler 裡直接可見 — Nsight Compute 的 `l1tex__data_bank_conflicts_pipe_lsu_mem_shared` counter, 或 omniperf 的 LDS bank conflict 指標。

## Matrix transpose 三部曲

**Step 1: naive** — 直接 `out[x*n + y] = in[y*n + x]`。 讀是 coalesced, 但寫是 strided, global 存取效率很差。

**Step 2: shared memory** — 先把一個 tile 讀進 shared memory, 再寫出, 讓讀寫都 coalesced。 但若 tile 宣告成 `tile[32][32]`, 轉置讀取時整個 warp 落在同一 column → 同一 bank → 32-way conflict。

**Step 3: conflict-free padding** — 把 tile 宣告成 `tile[32][33]`, 多一個 padding column, 錯開 bank 對映, conflict 消失。

<PlatformTabs>
<Platform name="CUDA">

```cpp
// Padded tile: the +1 column shifts each row's bank mapping, removing the
// 32-way conflict that a [32][32] tile suffers on the transposed read.
constexpr int kTile = 32;
__global__ void transposePadded(const float* in, float* out, int n) {
    __shared__ float tile[kTile][kTile + 1]; // +1 padding avoids bank conflicts
    int x = blockIdx.x * kTile + threadIdx.x;
    int y = blockIdx.y * kTile + threadIdx.y;
    if (x < n && y < n) tile[threadIdx.y][threadIdx.x] = in[y * n + x];
    __syncthreads();
    int tx = blockIdx.y * kTile + threadIdx.x;
    int ty = blockIdx.x * kTile + threadIdx.y;
    if (tx < n && ty < n) out[ty * n + tx] = tile[threadIdx.x][threadIdx.y];
}
```

<Callout variant="warn" title="padding 不是免費午餐">
`[32][33]` 多用了一整欄的 shared memory, 可能壓低 occupancy。 多數情況這筆交易划算, 但仍要量測 — 有時 swizzle (用 XOR 重排 index) 比 padding 更省容量。
</Callout>

## 進階替代: XOR swizzle

Padding 用容量換 conflict-free; swizzle 用一點 index 計算換到同樣效果, 不浪費任何容量:

```cpp
// XOR swizzle: permute the column index by the row index. Rows keep distinct
// bank mappings without any padding column.
__shared__ float tile[kTile][kTile];
int col = threadIdx.x ^ threadIdx.y;          // swizzled column on store
tile[threadIdx.y][col] = in[y * n + x];
__syncthreads();
int rcol = threadIdx.y ^ threadIdx.x;         // apply the same permutation on load
float v = tile[threadIdx.x][rcol];
```

XOR 是一個 permutation, 每一 row 的 bank 對映都被錯開, 且讀寫兩邊用同一條公式即可還原。 CUTLASS 與高效能 GEMM kernel 內部大量使用這類 swizzle (第 12、14 章的 shared memory layout 都是變體)。 手寫小 kernel 用 padding 就夠; 寫 tile 很大、shared memory 吃緊的 kernel 時, swizzle 是標準答案。

## 其他你該知道的細節

- **寬資料型別**: 存取 `double` 或 `float2` (8 bytes) 時, 一個元素橫跨兩個 bank, conflict 規則以 bank 為單位重新計算; NVIDIA 部分架構支援 64-bit bank 模式。`float4` (16 bytes) 的 shared memory 存取由硬體拆成多個 phase, 各 phase 分別判定 conflict。
- **Broadcast 是免費的**: 整個 warp 讀同一個 address (例如讀 tile 的同一個係數) 由 broadcast 機制一次服務, 不算 conflict。 多個「部分」broadcast (多組 threads 各讀幾個不同 address) 稱為 multicast, 也高效。
- **動態 shared memory**: 大小要到 launch 時才知道時, 用 `extern __shared__ float smem[];` 並在 launch 的第三個參數傳 bytes。 超過 48 KB (NVIDIA) 需要 `cudaFuncSetAttribute` 明確 opt-in。

## Shared Memory Design Checklist

設計 shared memory tile 時, 逐項檢查:

- **Global load phase**: threads 是否連續讀 global memory? 如果不是, shared memory 只是把 uncoalesced load 搬到前面。
- **Shared layout**: compute 階段的主要讀取方向是否 conflict-free? GEMM 常為 A/B 分別設計不同 layout。
- **Capacity**: tile、padding、double buffer、epilogue scratch 加總後是否壓低 occupancy?
- **Synchronization**: 每一輪 producer 寫完 shared memory 後, consumer 是否有 barrier? double buffer 是否避免覆寫尚未讀完的 stage?
- **Replacement option**: 資料只在 warp 內共享時, shuffle 可能比 shared memory 更快。

Stencil 是另一個好例子: block 載入 central tile 加 halo 到 shared memory, 讓相鄰 output 重用輸入。 若 halo 載入分支太多, 可讓每個 thread 先載 central element, 再由邊界 threads 補 halo; 若 tile 寬度剛好造成 bank alias, padding 一欄通常比忍受 conflict 划算。
