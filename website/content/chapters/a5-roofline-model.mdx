# Roofline Model

你花了一週優化一個 kernel, 從 200 GFLOP/s 推到 400。 這算好嗎? 如果它的天花板本來就是 15000, 你才跑到 3%; 如果天花板是 420, 你已經榨乾了。 沒有 roofline model, 你根本不知道自己站在哪。 這一章教你用一張圖判斷任何 kernel 的優化上限。

## Arithmetic intensity 是關鍵座標

Roofline 的橫軸是 arithmetic intensity (AI): 每從 memory 搬一個 byte, 做了多少 FLOP。

$$
\text{AI} = \frac{\text{FLOP count}}{\text{bytes moved}} \quad (\text{FLOP/byte})
$$

縱軸是可達到的 throughput (GFLOP/s)。 屋頂由兩段組成: 一條斜線 (受 memory bandwidth 限制,$\text{perf} = \text{BW} \times \text{AI}$) 和一條水平線 (受 compute peak 限制)。 兩者的交點叫 ridge point。

$$
\text{attainable perf} = \min (\text{peak compute},\; \text{peak bandwidth} \times \text{AI})
$$

## Bandwidth-bound vs. compute-bound

- AI 落在 ridge point **左邊** → 你在斜線下, 是 **bandwidth-bound**: 瓶頸是搬資料。 優化方向是提高 coalescing、減少冗餘搬移、提高 reuse (拉高 AI)。
- AI 落在 ridge point **右邊** → 你在水平線下, 是 **compute-bound**: 瓶頸是算術單元。 優化方向是提高 ILP、用 FMA、用 Tensor Core/MFMA。

## 常見 Kernel 的 AI 落點

建立幾個錨點, 以 fp32、不做任何 reuse 優化的 naive 實作估算:

| Kernel | FLOP / element | Bytes / element | AI (FLOP/byte) | 定性 |
| --- | --- | --- | --- | --- |
| Copy / transpose | 0 | 8 | 0 | 純 bandwidth |
| Vector add | 1 | 12 | ~0.08 | 深度 bandwidth-bound |
| SAXPY | 2 | 12 | ~0.17 | 深度 bandwidth-bound |
| Naive GEMM | 2K per output | 8K per output | 0.25 | bandwidth-bound (零 reuse) |
| Tiled GEMM (tile = T) | — | — | O (T) | 隨 tile 增大爬向 compute roof |

注意 naive GEMM 的 AI 與矩陣大小無關 — 不做 reuse, 它永遠是 0.25。 Tiling 之所以是 GEMM 優化的第一課 (第 12 章), 就是因為只有它能改變 AI 本身。

<Callout variant="insight" title="roofline 給你的第一個問題">
每次看到一個慢 kernel, 先問: 「它是 bandwidth-bound 還是 compute-bound?」這一個判斷就決定了你接下來所有的優化方向。 搞錯方向, 你可能花一週優化 compute, 而瓶頸其實在 memory。
</Callout>

## 進階: 階層式 Roofline 與量測值

兩個實務上的細節, 讓 roofline 從示意圖變成診斷工具:

- **理論 AI vs. 量測 AI**: 演算法算出來的 bytes 是「理想搬運量」; 實際搬運量受 coalescing 與 cache 影響, 可能大得多。 用 profiler 的 DRAM bytes 算出「量測 AI」, 若遠低於理論值, 代表 memory 系統在做白工 (第 6 章)。
- **階層式 roofline**: 對每一層 memory (L1、L2、HBM) 各畫一條斜線。 一個 kernel 可能對 HBM 是 compute-bound、對 L2 卻是 bandwidth-bound — 這告訴你瓶頸在哪一層。 Nsight Compute 的 roofline section 可以直接畫出來。

還有一個隱藏假設要記得: roofline 假設 memory 與 compute 完全重疊。 若你的 kernel 因為 latency 沒被藏好而兩者都閒著 (latency-bound), 它會遠低於兩條屋頂 — 這種情況先回第 8 章補 occupancy 或 ILP, 再談 roofline。

## 怎麼把你的 Kernel 標上去

1. 數出 FLOP count (演算法決定, 通常固定)。
2. 估 bytes moved (理想值可從演算法算, 實際值可從 profiler 的 memory throughput 讀)。
3. AI = FLOP / bytes, 量到的 GFLOP/s 當縱座標, 點上去。
4. 看點離哪一段屋頂近, 決定優化方向。

## Optimizations Move The Point

Roofline 的好處是能把技巧翻譯成座標變化:

- **Coalescing**: 不改理論 AI, 但降低實際 DRAM bytes, 讓量測 AI 接近理想值, 點往右上移。
- **Tiling / caching**: 增加每 byte 的 reuse, 直接提高理論 AI。 GEMM 從 naive 到 shared/register tiling 就是典型。
- **Fusion**: 移除中間 tensor 的 global write/read, bytes 下降, AI 上升; 但 register pressure 可能把點往下拉。
- **Vectorization**: bytes 不變, 但指令 overhead 和 memory transaction 效率改善, 點主要往上移。
- **Tensor Core / MFMA**: compute roof 變高, 但只有 AI 足夠高時才吃得到; memory-bound kernel 換低精度也可能只是降低 bytes。
- **Quantization / compression**: bytes 下降, AI 上升, 但要把 scale metadata 和 dequant FLOP 算進去。

所以每次優化後都要重算一次點。 當 bandwidth-bound kernel 被 tiling 推過 ridge point, 下一步就不再是減 bytes, 而是提高 compute pipe utilization。
