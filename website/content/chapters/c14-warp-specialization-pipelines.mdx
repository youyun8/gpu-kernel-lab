# Warp Specialization 與 Producer–Consumer Pipelines

到目前為止, 我們的 kernel 裡每個 warp 都做一樣的事。 但現代高效能 GEMM 與 attention kernel 不是這樣寫的: 它們讓一部分 warp 專門搬資料 (producer), 另一部分專門算數 (consumer), 像工廠的生產線一樣把 memory 與 compute 完全重疊。 這一章解釋這個結構。

## 為什麼要 Warp Specialization

Double buffering (第 12 章 Step 4) 已經讓 load 與 compute 重疊, 但所有 warp 還是既搬又算, 受限於同一組資源與同步點。 Warp specialization 更徹底:

- **Producer warps** 只負責把下一批 tile 從 global memory 非同步搬進 shared memory。
- **Consumer warps** 只負責從 shared memory 讀資料餵給 Tensor Core / MFMA。
- 兩組之間用 shared memory 上的 circular buffer 與輕量同步 (barrier / named barrier) 協調。

這讓 memory pipeline 和 compute pipeline 幾乎獨立運轉, latency 被藏得更乾淨, 尤其在 K 很大的深度 pipeline 上。 還有一個隱藏好處: producer 與 consumer 各自需要的 register 不同 (producer 少、consumer 多), Hopper 上可以用 `setmaxnreg` 動態重分配 register, 讓 consumer warp 拿到更大的 accumulator 空間 — 這是「同質 warp」做不到的。

## 非同步複製: cp.async 與 TMA

傳統 load 會佔住 register 與 warp, 直到資料到齊。 **非同步複製**讓 warp 發出「把這段 global 搬到 shared」的請求後就繼續做別的, 資料到了再用 barrier 確認。

- **NVIDIA `cp.async`** (Ampere+): global → shared 的非同步拷貝, 資料不經過 register file, 是 software pipelining 的硬體基礎。
- **NVIDIA TMA** (Hopper+): Tensor Memory Accelerator, 由專用硬體搬整塊多維 tile。 你預先建好 tensor map 描述子, kernel 內由**單一 thread** 發出一次 TMA 指令, 連 address 計算與邊界處理都卸載掉; 完成通知走 `mbarrier` (arrive/wait 語意的非同步 barrier)。
- **AMD 對應機制**: CDNA 透過 direct-to-LDS load 路徑 (`buffer_load ... lds`) 讓資料繞過 register 直達 LDS, 用 `s_waitcnt` 計數器追蹤未完成的 memory op; CDNA3 之後的 async copy 能力持續增強。 概念一致: 發出、繼續做事、需要時等計數歸零。

<Callout variant="insight" title="心智模型">
把 kernel 想成一條裝配線: producer 不斷把原料 (tile) 放上輸送帶 (shared memory circular buffer), consumer 從另一端取料加工 (MMA)。 只要輸送帶不空、不滿, 兩端都不會閒等 — 這就是 producer–consumer pipeline 的目標。 Stage 數就是輸送帶的長度: 越深越能吸收 latency 抖動, 但吃越多 shared memory。
</Callout>

## 多 Stage Pipeline 的節奏

```cpp
// Multi-stage async pipeline (Ampere cp.async). kStages buffers in shared
// memory form a circular queue between the copy engine and the math loop.
constexpr int kStages = 3;
__shared__ float sA[kStages][kTile][kTile];

// Prologue: fill all but one stage.
for (int s = 0; s < kStages - 1; ++s) {
    __pipeline_memcpy_async(&sA[s][0][0], &gmem[tileOffset(s)], kTileBytes);
    __pipeline_commit();
}
for (int t = 0; t < numTiles; ++t) {
    // Wait until the oldest in-flight copy (stage t % kStages) has landed.
    __pipeline_wait_prior(kStages - 2);
    __syncthreads();
    computeTile(sA[t % kStages]);            // MMA on the ready stage
    // Refill the stage we just consumed with a future tile.
    int next = t + kStages - 1;
    if (next < numTiles) {
        __pipeline_memcpy_async(&sA[next % kStages][0][0],
                                &gmem[tileOffset(next)], kTileBytes);
        __pipeline_commit();
    }
}
```

Hopper 的 warp-specialized kernel 把這個節奏再拆開: producer warp 專跑 TMA 發射迴圈, consumer warpgroup 專跑 `wgmma` (warpgroup 層級的 MMA, 4 個 warp 共同運算一大塊), 兩邊靠 mbarrier 的 phase 交替同步 — 這就是 CUTLASS 3.x 與 FlashAttention-3 的 kernel 骨架。

這些 pipeline 手寫極其容易出錯 (相依、buffer 管理、corner case), 因此實務上多半交給 CUTLASS / Composable Kernel — 但理解結構讓你讀得懂它們、也 debug 得動。

## 一個判斷: 你的 Kernel 需要這一套嗎?

Warp specialization 的收益來自「memory 與 compute 都接近飽和、卻互相卡同步點」的場景 — 大 GEMM、attention、深 K 的 pipeline。 如果你的 kernel 是 memory-bound 的 elementwise (compute 遠未飽和), 或小到填不滿機器 (第 13 章), 這套結構只會增加複雜度。 先用 profiler 確認: compute pipe 使用率高、但 stall 集中在 barrier/等資料, 才是它的適應症。

## Pipeline Failure Modes

多 stage pipeline 常見 bug 和性能問題:

- **Stage overwrite**: producer 覆寫 stage 前, consumer 還沒讀完。 需要 phase bit、barrier 或 wait group 保證。
- **Insufficient stages**: load latency 大於 compute window, consumer 仍然等資料。 增加 stage 或加大 tile compute。
- **Too many stages**: shared memory 暴增, occupancy 掉太低, 反而沒有足夠 CTA 隱藏 tail。
- **Producer underutilized**: 只用一個 warp 搬資料但 address arithmetic 太重, copy engine 餓住。 TMA 可把多維地址計算卸載掉。
- **Consumer register pressure**: accumulator 太大導致 spill。 Hopper 的 warp specialization 可搭配 register redistribution, 但仍需看 ptxas 報告。
- **Barrier stall**: producer/consumer 工作量不平衡, 雙方在 barrier 互等。 調整 producer warp 數、tile shape 或 prefetch distance。

Debug 這類 kernel 時, 先縮成兩個 stage 和小 shape 做 correctness, 再逐步加 stage、加 async copy、加 warp specialization。 一次把所有硬體特性打開, 很難定位錯誤。

<Solution>
預期結果與判讀: async 版快於同步版, 差距取決於「load latency 佔 compute window 的比例」— compute 越輕、tile 越大, overlap 的收益越明顯。 讀程式時盯三個位置: (1) prologue 先填 kStages − 1 個 buffer (輸送帶先上料); (2) 主迴圈的 `__pipeline_wait_prior(kStages - 2)` 只等「最舊那筆」到齊, 而不是等全部 — 這就是 pipeline 與「每次都同步等待」的本質差異; (3) 消費完立刻補發下一筆 async copy, 讓 in-flight 數量恆定。 把 kStages 改成 1 重跑, 你會得到接近同步版的時間 — stage 深度就是 latency 的緩衝池。
