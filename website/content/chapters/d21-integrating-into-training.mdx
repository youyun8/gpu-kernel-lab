# 整合回訓練/推論流程

你的 custom kernel 在 microbenchmark 上贏了, 但要真正用在訓練或推論裡, 還有一段路: 它得被 `torch.compile` 認得、在多 GPU 下正確、而且未來別人改動時不會悄悄變慢。 這一章講「讓 custom op 成為一等公民」的工程實務。

## 用 torch.library 註冊 Custom Op

直接呼叫 extension 函式雖然能動, 但 `torch.compile` 看不懂它、也無法追蹤梯度或做 shape 推導。 用 `torch.library` 把它註冊成正式的 op, 並提供 `FakeTensor` (meta) 實作讓 compiler 能在不執行的情況下推 shape:

```python
import torch

# Register a proper custom op so torch.compile can trace and reason about it.
@torch.library.custom_op("mylib::gelu", mutates_args=())
def gelu(x: torch.Tensor) -> torch.Tensor:
    return ext.gelu_forward(x.contiguous())

@gelu.register_fake
def _(x):
    # Shape/dtype propagation without running the kernel (needed by torch.compile).
    return torch.empty_like(x)

# Register backward via the autograd dispatch key or setup_context/backward.
```

註冊完別急著上線, 先跑 `torch.library.opcheck(gelu, (x,))` — 它自動檢查 fake 實作、autograd、以及 op 在各 dispatch key 下的一致性, 把「compile 時才爆的錯」提前到測試期。

## 與 torch.compile 共存

註冊為 custom op 後, `torch.compile` 會把它當成一個不可分割的節點: 它不會嘗試 fuse 進去 (那是你的 kernel 內部的事), 但會正確地在它前後 fuse 其他 op、追蹤相依、規劃記憶體。 避免用會導致 graph break 的寫法 (例如 op 內部呼叫 `.item()` 或依 data-dependent 的 Python 控制流), 否則 compile 的效益會被打斷。

用 `TORCH_LOGS=graph_breaks python train.py` 檢查你的 op 有沒有引入 graph break;`torch._dynamo.explain` 能列出完整的 break 原因清單。 另一個相容性維度是 CUDA Graph (`mode="reduce-overhead"`): 你的 kernel 必須「capture-safe」— 不在 kernel 路徑上做 host 同步、不配置新記憶體 (用 PyTorch 的 allocator 拿 workspace), 否則會讓整個 graph capture 失敗。

## 多 GPU 注意事項

<Callout variant="warn" title="多 GPU 三個地雷">
1. **device guard**: kernel 內務必用 `CUDAGuard(x.device())`, 否則在非 default device 的 tensor 上會 launch 到錯的 GPU。
2. **current stream**: 用 `getCurrentCUDAStream()`, 尤其和 DDP/NCCL 的 communication stream 並存時, 錯誤的 stream 會造成 race 或死結。
3. **同步邊界**: custom op 若內部有跨 device 或 host 同步, 會拖垮整個 data-parallel 的 pipeline; 盡量保持純 device、非同步。
</Callout>

訓練場景還有兩個容易忽略的整合點: **混合精度** — autocast 下你的 op 會收到 half/bf16 輸入, 要嘛用 dispatch 支援 (第 18 章), 要嘛註冊 autocast 行為 (升精度執行); **determinism** — 若 kernel 用了 atomicAdd (第 10 章), 在 `torch.use_deterministic_algorithms(True)` 下應該報錯或提供確定性 fallback, 否則會破壞使用者的重現性假設。

## Regression benchmark: 防止效能悄悄退化

效能不是一次達標就永遠有效。 程式碼演進、library 升級、shape 改變, 都可能讓你的 kernel 悄悄變慢。 把 benchmark 做成 CI 可跑的 regression 測試:

```python
# Pin a performance budget; fail CI if the kernel regresses beyond tolerance.
def test_gelu_perf_regression():
    t = benchmark_ms(lambda: gelu(x))       # warmup + CUDA events (chapter 4)
    baseline_ms = load_baseline("gelu")     # recorded on this hardware
    assert t <= baseline_ms * 1.10, f"regression: {t:.3f} vs {baseline_ms:.3f} ms"
```

關鍵: baseline 綁定硬體 (不同 GPU 數字不同)、用穩健的計時 (warmup + events + 多次取中位數)、留合理容忍度避免 flaky。 除了絕對時間, 也保留一條「相對 torch.compile / 原生實作」的比值測試 — PyTorch 升級後若內建實作追上來了, 這條測試會提醒你: 也許該功成身退, 刪掉 custom kernel 回歸內建 (維護成本歸零也是一種優化)。

## Deployment Checklist

上線前逐項確認:

- **Dispatch**: CUDA、ROCm、CPU fallback、unsupported dtype/shape 的錯誤訊息都清楚。
- **Compile**: `torch.compile` 無 graph break, fake/meta 實作和真 kernel shape 一致。
- **Autograd**: backward、double backward 是否支援要明確, `gradcheck` 覆蓋 tail shape。
- **AMP**: autocast 下 dtype 行為符合模型預期。
- **Determinism**: deterministic mode 下提供 fallback 或明確報錯。
- **Distributed**: DDP/FSDP 下 device guard、stream、allocator 和 communication overlap 不被破壞。
- **Observability**: benchmark、profiler trace、版本資訊、tuning cache 都能追溯。

這份 checklist 的目的不是增加流程, 而是避免 custom kernel 成為訓練系統裡最難 debug 的黑盒。
