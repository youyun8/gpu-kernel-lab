# Triton 入門與比較

手寫 CUDA/HIP 讓你完全掌控, 但每一行 shared memory 管理、每一個 index 計算都是你的責任。 Triton 換一個交易: 你用 Python 寫 block-level 邏輯, compiler 負責 coalescing、shared memory、pipelining。 這一章用 softmax 與 GEMM 重寫, 直接感受 productivity 與 peak performance 的取捨。

## Triton 的心智模型

在 Triton, 你不寫「每個 thread 做什麼」, 而是寫「每個 program (≈ block) 處理哪一塊資料」。 用 `tl.load` / `tl.store` 搬 block, 用 `tl.arange` 產生 index, compiler 把它降階成高效的 GPU 指令。

```python
import triton
import triton.language as tl

@triton.jit
def softmax_kernel(x_ptr, y_ptr, n_cols, stride, BLOCK: tl.constexpr):
    # One program handles one row. Numerically stable softmax (subtract max).
    row = tl.program_id(0)
    cols = tl.arange(0, BLOCK)
    mask = cols < n_cols
    ptr = x_ptr + row * stride + cols
    x = tl.load(ptr, mask=mask, other=-float("inf"))
    x = x - tl.max(x, axis=0)
    num = tl.exp(x)
    y = num / tl.sum(num, axis=0)
    tl.store(y_ptr + row * stride + cols, y, mask=mask)
```

對照第 10 章手寫的 softmax: 同樣的 numerical stability (減 max)、同樣的 reduction, 但你沒寫任何 `__shared__`、`__syncthreads` 或 shuffle。 Compiler 幫你處理了。

幾個對應關係值得記住: `program` ≈ CTA、`BLOCK` 這類 `tl.constexpr` 參數 ≈ 編譯期 tile 尺寸、`mask` ≈ 邊界檢查。 你在前面章節建立的所有概念 (tiling、coalescing、reduction) 都還在, 只是換成 block 粒度表達。

## Autotune: 讓 Triton 自己找 config

Tile 尺寸、warp 數、pipeline stage 數的最佳值依 shape 與硬體而異 (第 13、15 章)。 Triton 內建 autotuner, 宣告候選 config, 首次遇到新 key 時實測挑最快:

```python
@triton.autotune(
    configs=[
        triton.Config({"BM": 128, "BN": 128, "BK": 32}, num_warps=8, num_stages=3),
        triton.Config({"BM": 64, "BN": 128, "BK": 32}, num_warps=4, num_stages=4),
        triton.Config({"BM": 64, "BN": 64, "BK": 64}, num_warps=4, num_stages=4),
    ],
    key=["M", "N", "K"],   # re-tune when these change
)
@triton.jit
def matmul_kernel(a_ptr, b_ptr, c_ptr, M, N, K, ...):
    ...
```

這正是 library autotune (第 15 章) 的輕量版: 冷啟動付一次搜索成本, 之後同 shape 直接命中 cache。 Benchmark Triton kernel 時務必先 warmup 到 autotune 收斂, 否則量到的是搜索時間。

## Productivity vs. peak performance

<Callout variant="insight" title="這個取捨怎麼看">
Triton 讓你在幾十行內寫出「接近手調」的 kernel, 開發速度快一個數量級。 代價是最後那幾 % 的 peak — 對極致場景, 手調 CUTLASS 仍可能勝出, 因為它能觸及 Triton 抽象暫時觸不到的硬體細節 (某些 TMA/swizzle 模式)。 但對絕大多數自訂 op, Triton 的「夠快 + 好維護」是更划算的選擇。
</Callout>

GEMM 也一樣: Triton 官方的 matmul tutorial 用 autotune 掃 tile size, 在大矩陣上能到 cuBLAS 的八九成, 而程式碼比手寫 CUTLASS 短得多。

想看 compiler 到底做了什麼, 可以檢視 `matmul_kernel.cache` 目錄下生成的 PTX/AMDGCN, 或用 `TRITON_PRINT_AUTOTUNING=1` 觀察 autotune 過程 — 對照第 11 章「驗證你的假設」的紀律, DSL 也一樣適用。

## CUDA 與 ROCm 可攜性

Triton 同時支援 NVIDIA 與 AMD backend: 多數情況同一份 `.py` 不改就能兩邊跑, autotuner 各自找該平台的最佳 config。 這是相對手寫 CUDA (需 hipify) 或分別維護兩份 kernel 的一大優勢。 要注意的是, peak 效能與可用的進階特性 (某些 MMA 模式) 仍可能因 backend 成熟度而異。

也因此 Triton 成了整個 PyTorch 生態的可攜性基石: `torch.compile` 的 Inductor backend 生成的就是 Triton kernel, 一份 codegen 邏輯同時服務 CUDA 與 ROCm; vLLM/SGLang 的許多自訂 op (量化、MoE routing、sampling) 也選 Triton 實作, 理由相同 (第 24、25 章)。

## Triton Performance Levers

Triton 仍需要你做 kernel 工程判斷:

- **BLOCK size**: 太小 overhead 高, 太大 register/shared memory 壓力高。 Softmax 常選 next power of two, 但長 row 要分塊。
- **num_warps**: reduction-heavy kernel 通常需要更多 warps; 小 block 用太多 warps 反而浪費。
- **num_stages**: matmul/pipeline 用 stage 隱藏 memory latency, 但 stage 越多 shared/register 壓力越高。
- **mask cost**: 邊界 mask 太多會拖慢 hot path。 固定常見 shape 時可用 specialized kernel 避免多餘 mask。
- **program order**: 改 `pid` 到 tile 的 mapping 可以改善 L2 locality, 等同 CTA swizzle。
- **cache modifiers**: `tl.load(..., cache_modifier=".cg")` 這類 hint 只在你確定 reuse 距離時使用。

把 Triton 當成「幫你管理 thread 細節的 kernel language」, 而不是自動最佳化魔法。 Profiler 看到的 coalescing、occupancy、L2 hit、register spill, 仍然用前面章節的方式解讀。
