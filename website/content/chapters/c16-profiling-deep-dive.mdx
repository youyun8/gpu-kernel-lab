# Profiling 深入

Profiler 吐出上百個 metric, 新手看了只會更焦慮。 但其實你不需要全懂 — 你需要的是一套「從症狀回推病因」的流程。 這一章教你怎麼讀 Nsight Compute / omniperf 的關鍵 section, 以及一個能反覆套用的 bottleneck 診斷 workflow。

## 先分類: Memory-Bound 還是 Compute-Bound

一切從 speed-of-light / roofline section 開始。 Nsight Compute 的 GPU Speed of Light section 直接給你 Compute (SM) 與 Memory 兩個 throughput 百分比:

- Memory 高、Compute 低 → **memory-bound**, 往 memory workload analysis 深入。
- Compute 高、Memory 低 → **compute-bound**, 往 scheduler / instruction 深入。
- 兩者都低 → **latency-bound**, 往 scheduler statistics 找 stall 原因。

指令長這樣 — 這兩行就能回答「bound 在哪」:

```bash
# NVIDIA: speed-of-light + memory sections for one kernel, skipping warmup.
ncu --section SpeedOfLight --section MemoryWorkloadAnalysis \
    -k regex:sgemm --launch-skip 10 --launch-count 1 ./sgemm_step0_naive

# AMD: kernel stats first (find the hot kernel), then the full analysis.
rocprofv3 --stats ./sgemm_step0_naive
omniperf profile -n step0 -- ./sgemm_step0_naive && omniperf analyze -p workloads/step0
```

## 三個關鍵 section 怎麼讀

**Memory Workload Analysis:** 看 L1/L2 hit rate、DRAM throughput、每個 request 的 sector 使用率。 Sector 使用率低 = coalescing 差 (回第 6 章); L2 hit 低 = locality 差 (考慮 swizzle, 第 13 章)。

**Scheduler Statistics / Warp State:** 看每個 issue slot 的 stall 原因。 常見的 `Stall Long Scoreboard` = 在等 global memory, 代表 latency hiding 不足 (加 warp 或做 prefetch)。 `Stall MIO Throttle` = shared memory/特殊功能單元壅塞。 `Stall Barrier` = 在等 `__syncthreads()`, 常見於 block 內工作不均。 `Stall Wait` / `Stall Short Scoreboard` = 短相依鏈, 通常還好。

**Occupancy Section:** 顯示 achieved vs. theoretical occupancy, 以及 limiter。 Achieved 遠低於 theoretical 通常是 load imbalance 或 tail effect (第 13 章)。

## 常見「症狀 → 病因 → 章節」對照表

| 症狀 (metric) | 最可能病因 | 回哪一章 |
| --- | --- | --- |
| sectors/request 遠高於理想 | coalescing 差、AoS 佈局 | 第 6 章 |
| shared memory bank conflict counter > 0 | 存取 pattern 撞 bank | 第 7 章 |
| Stall Long Scoreboard 佔比高 | latency hiding 不足 | 第 8、14 章 |
| theoretical occupancy 低, limiter = registers | register 壓力 | 第 8、11 章 |
| achieved ≪ theoretical occupancy | wave quantization / 負載不均 | 第 13 章 |
| L2 hit rate 低且 DRAM 飽和 | CTA 排程沒有 locality | 第 13 章 |
| Tensor pipe 使用率低 (MMA kernel) | 佈局/pipeline 沒餵飽 MMA | 第 12、14 章 |

## Omniperf 的對應

AMD 的 omniperf (rocprof-compute) 提供類似的分層視圖: speed-of-light、memory chart (LDS、L1/L2/vL1D、HBM)、以及 wavefront / VALU / MFMA 使用率。 概念一一對應:

| 診斷目標 | Nsight Compute | omniperf |
| --- | --- | --- |
| bound 分類 | Speed of Light | System Speed-of-Light |
| memory 效率 | Memory Workload Analysis | Memory Chart |
| stall 原因 | Scheduler / Warp State | Wavefront / stall breakdown |
| 矩陣單元使用率 | Tensor pipe utilization | MFMA / VALU utilization |

## 進階技巧: 把 metric 釘到程式碼行

- **Source counters**: `ncu --set full` 搭配 `-lineinfo` 編譯 (nvcc 加 `-lineinfo`), Nsight Compute 的 Source page 能把 stall 與 memory 指標對應到每一行 C++/SASS — 直接看到「哪一行在等」。
- **只抓你要的 kernel**: `ncu -k regex:sgemm --launch-skip 10 --launch-count 1` 跳過 warmup、鎖定目標 kernel, 避免蒐集成本污染整個 run (ncu 的 replay 會讓程式慢數十倍)。
- **比較模式**: `ncu --import baseline.ncu-rep` 可以把兩個版本的報告並排 diff — 驗證「這次改動確實改善了目標 metric」的最快方式。
- **警惕觀測成本**: profiler 本身會改變時脈與 cache 行為; 最終數字仍以 benchmark harness (第 4 章) 為準, profiler 負責解釋「為什麼」。

把這幾招串起來就是一個可重複的診斷流程:

```bash
# 1. Compile with line info so the Source page can map stalls to C++ lines.
nvcc -O3 -lineinfo -o sgemm sgemm.cpp

# 2. Full-detail capture of the target kernel only.
ncu --set full -k regex:sgemm --launch-skip 10 --launch-count 1 \
    -o after_fix ./sgemm

# 3. Diff against the baseline capture to confirm the metric moved.
ncu --import baseline.ncu-rep --import after_fix.ncu-rep
```

## Systematic workflow

<Callout variant="insight" title="可反覆套用的六步驟">
1. 跑 speed-of-light, 分類 memory / compute / latency-bound。
2. 依分類進對應 section 找主要 stall 或低效率指標。
3. 提出一個假設 (例如「sector 使用率 25% → coalescing 差」)。
4. 只改一件事去驗證假設。
5. 重新 profiling, 確認該 metric 改善且整體變快。
6. 回到步驟 1, 直到接近 roofline 或收益遞減。
</Callout>

關鍵紀律: 一次只改一個變因, 並讓 profiler 而非直覺告訴你瓶頸在哪。 很多優化「感覺應該更快」卻沒效, 就是因為改錯了維度。

## Metric-To-Fix Examples

把 profiler metric 直接接到修改:

- **Global load sectors/request 高** → 改 SoA、padding、vectorized load 或重新安排 thread-to-data mapping。
- **L2 hit rate 低且 tile 有 reuse** → 試 CTA swizzle、larger tile、persistent scheduler 或 prefetch distance。
- **Shared bank conflict 高** → 加 padding、XOR swizzle、改 shared layout 或把最後階段改 shuffle。
- **Register spill** → 縮小 micro-tile、降低 unroll、拆 epilogue、或放寬 launch bounds。
- **Barrier stall 高** → 減少 `__syncthreads()`、改 warp-level primitive、平衡 block 內工作、或拆 residue path。
- **Atomic throughput/stall 高** → privatize、warp aggregate、sort + segment reduce 或兩階段 reduce。
- **Launch gap 高** → fuse、graph replay、persistent kernel 或把 framework graph break 消掉。

每個 fix 都要有對應 metric 下降。 若 wall time 變快但 metric 沒變, 你可能改善的是別的瓶頸; 若 metric 變好但 wall time 不變, 瓶頸已轉移。
