# Instruction-Level 優化

你的 kernel 已經 coalesced、沒有 bank conflict、occupancy 也合理, 但還差一截。 這時該往下看指令層級: 一次搬多少 bytes、迴圈展開幾次、乘加能不能合併成一條指令。 這些是壓榨最後 10–20% 的工具。

## Vectorized Memory Access (float4)

一個 load 指令搬 32-bit 和搬 128-bit 的 overhead 差不多, 但後者一次拿 4 個 float。 用 `float4` / `int4` 讓每個 thread 一次搬 128 bits, 減少指令數與 address 計算, 通常直接改善 memory-bound kernel。

```cpp
// Vectorized copy: each thread moves 128 bits per load/store instruction.
// Requires the pointer and count to be 16-byte aligned / divisible by 4.
__global__ void copyVec4(const float4* in, float4* out, int n4) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n4) out[i] = in[i];
}
```

<Callout variant="warn" title="對齊是前提">
`float4` 要求 16-byte 對齊。 對未對齊的 buffer 硬轉會 undefined behavior 或效能反效果。 確認 allocation 對齊、元素數是 4 的倍數, 或對尾端做 scalar fallback。
</Callout>

## Loop unrolling 與 #pragma unroll

展開迴圈可消除迴圈控制的 overhead、暴露更多獨立指令給 scheduler (提高 ILP)、也讓編譯器有機會做常數摺疊。 對已知次數的內層迴圈用 `#pragma unroll`:

```cpp
// Unrolling the inner accumulation exposes independent FMAs (more ILP) and
// removes loop overhead. Best when the trip count is a small compile-time value.
#pragma unroll
for (int i = 0; i < kUnroll; ++i) {
    acc[i] = fmaf(a[i], b[i], acc[i]);
}
```

過度展開會膨脹 register 使用、壓低 occupancy, 要量測拿捏。

一個容易忽略的細節: unroll 要真正產生 ILP, **累加器也要展開**。`for(...) sum += a[i] * b[i];` 就算展開 8 次, 8 條 FMA 仍串在同一個 `sum` 上, 彼此相依、無法重疊; 改用 `sum0..sum3` 四個獨立累加器、最後再相加, 才把相依鏈切短。 這對 latency-bound 的累加迴圈常有立竿見影的效果。

<Callout variant="insight" title="落到真實場景: SAXPY 的 unroll 甜蜜點是 2~4 不是 8">
把一支 memory-bound 的 SAXPY (`y = a*x + y`) 拿來掃 unroll factor, 你會親眼看到曲線先升後平再掉: unroll 2/4 提供夠多獨立的 load/store 把 memory latency 藏住, 時間下降; 推到 8/16 時 `-Xptxas=-v` 顯示 register 開始暴漲、occupancy 掉一級, 而 memory-bound kernel 根本不缺算術吞吐, 於是持平甚至變慢。 實務判法: 對每個 factor 記下 (時間, register 數), 時間持平但 register 開始跳的那一格就停 — 這是第 8 章「ILP 換 occupancy」的指令層版本。
</Callout>

## FMA 與 fast math 的取捨

`fmaf(a, b, c)` 把乘加合併成一條指令, 不只快一倍, 還少一次中間 rounding (精度反而更好)。 編譯器通常會自動合併, 但明確寫出可確保發生。

`--use_fast_math` (nvcc) / `-ffast-math` (hipcc) 把 `sinf`、`expf`、除法等換成低精度高速版本, 並放寬 IEEE 語意。 對容忍度高的 workload (很多 DL inference) 值得, 對需要精確結果的科學計算要謹慎。 也可以細粒度控制: 用 `__expf`、`__fdividef` 等 intrinsic 只在確定安全的位置換快速版本, 而不是整個檔案一刀切。

<Callout variant="warn" title="落到真實場景: 整個檔案開 --use_fast_math 炸了科學計算">
`--use_fast_math` / `-ffast-math` 對 DL inference 常常白賺速度 (`__expf`、fast divide 的誤差在容忍範圍內), 但它是**整個 translation unit 一刀切**: 它同時把 `sqrtf`、除法換成低精度版, 還放寬了 IEEE 語意 (例如假設沒有 NaN / Inf)。 如果同一個檔案裡混了一段需要精確的 reduction 或物理模擬, 結果會悄悄漂掉、甚至因為「編譯器假設不會有 NaN」而讓你的 NaN 檢查被最佳化掉。 安全作法: 不要全域開, 只在確定安全的熱點用 `__expf`、`__fdividef` 這類 intrinsic 精準換掉那幾個算式。
</Callout>

## Branch 與 predication

短小的 if-else, 編譯器常轉成 predicated instruction (兩邊都算、依條件選結果), 沒有真正的分支, divergence 成本趨近於零。 所以「一個小小的邊界檢查」不必焦慮。 真正貴的是: 兩個分支都很長、或迴圈次數 data-dependent 且 warp 內不一致。 優化的手段是在演算法層面讓同 warp 的工作同質 (排序、分桶), 而不是硬拗語法。

<Callout variant="insight" title="落到真實場景: 該擔心的 divergence 長什麼樣">
邊界檢查 `if (i < n)` 你不用怕 — 編譯器 predicate 掉了。 真正咬人的是 data-dependent 且 warp 內不一致的分支: 例如一個 warp 裡各 token 走不同專家 (MoE)、或各 sequence 長度不同讓迴圈次數天差地遠, 這時 warp 得把每條路徑輪流跑一遍, 有效吞吐被打對折以上。 解法不在語法而在演算法: 先按 expert id / 長度排序或分桶, 讓同一個 warp 的 work 同質, 再進 kernel。 這正是 MoE dispatch、sorted-batch attention 這些前處理存在的理由。
</Callout>

## 驗證你的假設: 看編譯器輸出

指令層級的優化最忌「以為」。 兩個工具讓你直接看事實:

```bash
# Register usage, spills, and static shared memory per kernel
nvcc -O3 -arch=sm_80 -Xptxas=-v kernel.cu

# Dump the actual machine code (SASS) to see FMA fusion, unrolling, etc.
cuobjdump -sass ./kernel | less

# AMD: emit ISA and resource usage
hipcc -O3 --save-temps kernel.cpp   # inspect the .s file
```

也可以把程式碼貼進 Compiler Explorer (godbolt.org, 支援 nvcc/PTX 與 hipcc/GCN ISA), 即時看每行 C++ 對應的機器碼。 確認三件事: FMA 有沒有合併、迴圈有沒有真的展開、有沒有意外的 spill (`STL`/`LDL` 指令)。

<Callout variant="insight" title="優化順序的紀律">
指令層級是「最後一哩」: 先確定 memory access pattern 與演算法已經對了 (第 6–10 章), 再來摳指令。 一個 coalescing 沒做好的 kernel, unroll 得再漂亮也是白工 — profiler 說瓶頸在哪, 才往哪裡使力。
</Callout>

## Micro-Optimization Checklist

摳指令前先列清楚假設:

- **Address arithmetic**: 內層 loop 中能否把乘法轉成 pointer increment? `row * stride + col` 不要每次重算。
- **Invariant hoisting**: scale、stride、base pointer、loop bound 能否搬到 loop 外?
- **Restrict and const**: `const float* __restrict__ a` 讓編譯器更敢重排 load。
- **Inline small helpers**: device function 太多層可能妨礙 unroll, 但過度 inline 會膨脹 code size。
- **Approximate math**: `rsqrt.approx`、`__expf`、fast divide 只放在誤差可控的位置。
- **Spill audit**: 每次加 unroll/vectorization 後都看 register 和 local memory load/store。

例如 normalization kernel 中, `1.0f / sqrtf(var + eps)` 可先算成 `inv_std`, 再讓所有元素乘 `inv_std`; 若每個元素都做一次 sqrt/divide, 指令層級再怎麼調都浪費。

<Callout variant="insight" title="落到真實場景: __restrict__ 讓 GEMM 內迴圈敢重排 load">
一支手寫 GEMM 的內層迴圈, 加不加 `const float* __restrict__` 差很多: 沒有 restrict, 編譯器必須假設 A、B、C 可能指向重疊記憶體, 於是每次都乖乖「load → 用 → 再 load」不敢把 load 提前或合併; 標上 restrict 後它才敢一次預抓多筆、把 address 計算提到迴圈外、暴露更多 ILP。 看 SASS 或 `-Xptxas=-v` 就能驗證: 加了之後 load 指令被重排、甚至 register 用量下降。 這是「只要你確定指標不重疊就零風險」、該預設加上的一行。
</Callout>
