# Occupancy 與 Latency Hiding

新手常把 occupancy 當成唯一的效能旋鈕: 「把 occupancy 拉到 100% 就對了」。 但很多世界級的 GEMM kernel 只跑在 25–50% occupancy, 卻逼近硬體極限。 這一章解釋 occupancy 到底是什麼、它為什麼重要、以及為什麼「越高不一定越快」。

## Occupancy 是什麼

Occupancy = 一個 SM/CU 上實際活躍的 warp 數 ÷ 硬體支援的最大 warp 數。 它衡量的是「你有多少 warp 可以拿來做 latency hiding」。 當一個 warp 卡在 memory, scheduler 需要別的 ready warp 來填空檔; warp 越多, 越容易把 latency 藏起來。

Occupancy 受三個資源限制, 取最嚴格的那個 (limiter):

- **Registers per thread** × block 大小 vs. register file 容量
- **Shared memory per block** vs. SM 的 shared memory 容量
- **Block / warp 數量上限** (硬體固定)

## 需要多少 warp 才夠? Little's law 的估算

「夠不夠」可以粗略地用排隊理論估: 要填滿一條 latency 為 $L$ cycles、每 cycle 可發射 $T$ 筆的 pipeline, 需要同時在飛的請求數約為 $L \times T$。 以 global memory latency ~500 cycles 為例, 每個 warp 若一次只發一筆 load, 你就需要幾十個 warp 輪流頂著; 但若每個 warp 一次發 4 筆獨立的 load (ILP), 需要的 warp 數就除以 4。 這條算式就是「occupancy 與 ILP 可互相替代」的數學根據。

## 用 API 查詢與驗證

不用猜, 編譯器和 runtime 會告訴你:

```cpp
// Ask the runtime how many blocks of this kernel fit per SM.
int numBlocks = 0;
cudaOccupancyMaxActiveBlocksPerMultiprocessor(&numBlocks, myKernel, 256, 0);
// HIP: hipOccupancyMaxActiveBlocksPerMultiprocessor(...)

// At compile time, ask the compiler to report per-kernel resource usage:
//   nvcc  -Xptxas=-v  kernel.cu     -> registers, spill stores/loads, smem
//   hipcc -Rpass-analysis=kernel-resource-usage ...
```

編譯時加 `-Xptxas=-v` 會印出每個 kernel 的 register 數、spill 與 shared memory 用量 — 出現 `spill stores` 就代表 register 壓力已經超標, 變數被踢到 local memory (第 3 章), 通常比降一點 occupancy 更傷。

## __launch_bounds__: 給編譯器目標

編譯器分配 register 時並不知道你想要的 occupancy。`__launch_bounds__` 告訴它上限, 讓它盡量把 register 壓在能達到目標 occupancy 的範圍內 (代價可能是 register spill 到 local memory)。

```cpp
// Hint the compiler: at most 256 threads/block, aim for >=2 blocks/SM.
// The compiler will try to cap register usage to honor this.
__global__ void __launch_bounds__(256, 2) myKernel(/* ... */) {
    // ...
}
```

HIP 的對應寫法相同 (`__launch_bounds__(256)`), CDNA 上另有 `__attribute__((amdgpu_waves_per_eu(n)))` 可以更直接地要求每個 execution unit 的 wave 數。

## 為什麼 higher occupancy 不一定更快

這是本章的核心洞見。 latency hiding 不只能靠「更多 warp」, 也能靠「單一 warp 內更多獨立指令」 (ILP)。 兩者是替代品:

- **走高 occupancy**: 很多 warp、每 thread 少 register、小 tile。 適合 memory-bound、每 thread 工作簡單的 kernel。
- **走低 occupancy + 大 tile + 高 ILP**: 少 warp、每 thread 用很多 register 算一大塊、指令間彼此獨立。 GEMM 這類 compute-bound kernel 常走這條路 — 大 tile 帶來高 reuse 與高 arithmetic intensity, 勝過多塞幾個 warp。

<Callout variant="insight" title="該記住的一句話">
occupancy 是手段不是目的。 真正的目的是「讓執行單元不閒著」。 當 ILP 已經足夠把 pipeline 餵飽, 再往上加 occupancy 只是浪費 register、甚至因為 tile 變小而降低 reuse。 用 profiler 看的是 stall 原因, 不是 occupancy 數字本身。
</Callout>

## Theoretical vs. achieved occupancy

Profiler 會報兩個數字: theoretical (資源計算的上限) 與 achieved (執行期實際平均)。 achieved 遠低於 theoretical 的常見原因: grid 太小填不滿機器 (第 13 章的 wave quantization)、block 之間工作量不均、kernel 尾端只剩少數 block 在跑。 這種情況再怎麼省 register 也沒用 — 問題在工作的「形狀」, 不在資源。

## Tuning Levers

Occupancy 相關調參不要只改一個 block size:

- **Block size**: 128、256、512 threads 是常見候選。 太小會浪費 scheduler/block slots, 太大會降低 block-level scheduling 彈性。
- **Register cap**: `__launch_bounds__` 或 compiler flag 可壓 register, 但一旦出現 spill 就要退回。
- **Tile size**: 大 tile 提高 reuse 和 ILP, 但吃 register/shared memory。 小 tile occupancy 高, 但可能 memory traffic 變多。
- **Independent accumulators**: 對 latency-bound loop, 多個 accumulator 可用 ILP 替代部分 occupancy。
- **Grid size**: achieved occupancy 低可能只是 blocks 太少。 小矩陣用 Split-K、batching 或 persistent scheduler 比調 register 有效。

一個 practical loop: 先用 profiler 看 stall reason。 若是 `long scoreboard`, 增加 active warps 或每 warp independent loads; 若是 `not selected` 很高, warp 已經足夠但 issue pipe 競爭, 該看 instruction mix; 若是 `math pipe throttle`, occupancy 不是瓶頸, 要調算術 pipeline 或使用 MMA。
