# Software Pipelining 是什麼

一條裝配線不會等第一台車完全組好才開始第二台: 車身、引擎、噴漆三個工站同時各忙一台不同的車。 **Software pipelining** 就是把這個想法搬到程式裡 — 把一段有多個 stage 的工作切開, 讓不同 stage 同時處理不同的資料, 用**重疊 (overlap)** 把各 stage 的 latency 藏在彼此後面。

這一章建立整個 track 的共同語言。 之後我們會看到同一個 idea 在六個不同的層級反覆出現: kernel 內的 double buffering、host–device 的 CUDA stream、PyTorch 的 data prefetch、跨 GPU 的 pipeline parallelism, 一直到 SGLang 推論服務的 overlap scheduler。

<Callout variant="insight" title="貫穿整個 Track 的一句話">
Pipelining 不會讓任何單一 stage 變快, 它讓多個 stage 同時忙。 只要相鄰 stage 用不同的硬體資源 (DMA 引擎 vs. SM、CPU vs. GPU、compute vs. network), 把它們重疊就等於免費拿到吞吐。
</Callout>

## 從序列到 pipeline

假設每個工作項要依序經過 3 個 stage (A→B→C), 每個 stage 花 1 個時間單位。 序列做法處理 N 項要 `3N` 步。 Pipeline 做法讓 stage A 一做完第 0 項就馬上開始第 1 項, 同時 stage B 處理第 0 項:

<PipelineConceptFigure />

處理 N 項、深度 k 的 pipeline 只需要 `N + k − 1` 步 (而非 `N · k`)。 當 N ≫ k, 平均每步就完成一項, 吞吐趨近**單一 stage 的速率**, 加速比趨近 k。

<Callout variant="insight" title="落到真實場景: 離線批次 embedding 的三 stage 管線">
這條 `N + k − 1` 式子最直接的應用是離線批次推論: 要把 100k 篇文件轉成向量, 每篇經過 tokenize (CPU) → GPU forward → 寫進向量 DB 三個 stage。 一篇一篇序列跑, GPU 大半時間在等 CPU tokenize 與 DB 寫入。 用一條 prefetch queue 讓 tokenizer 先備好後面的 batch、另一條 writer thread 背景 flush, 三個 stage 就重疊起來。 因為 N=100k ≫ k=3, fill/drain 可忽略, 整個 job 的 wall time 幾乎等於「最慢那個 stage 的總時間」(通常就是 GPU forward), CPU 與 I/O 開銷全被藏掉。
</Callout>

<Exercise id="sp1-e1" title="Pipeline 加速比" kind="paper" level="入門">
一個 4-stage pipeline, 每 stage 1 單位時間, 要處理 N 項。 (a) 序列版要幾步? (b) pipeline 版要幾步? (c) N=100 時 pipeline 相對序列的加速比? (d) N→∞ 時加速比趨近多少?

<Solution>
(a) 序列: `4N` 步。 (b) pipeline: `N + 4 − 1 = N + 3` 步。 (c) N=100: `400 / 103 ≈ 3.88×`。 (d) N→∞: `4N/(N+3) → 4×` (= stage 數 k)。 加速上限就是 pipeline 深度, 前提是各 stage 用不同資源、可真正並行。
</Solution>
</Exercise>

## Pipeline 的三個區間: fill、steady、drain

上圖左上角與右下角的斜三角就是 pipeline 沒滿載的地方:

- **Fill (暖機)**: 最前面 `k − 1` 步, 後段 stage 還沒拿到資料, 只有部分 stage 在忙。
- **Steady state (穩態)**: 中間, 所有 stage 同時滿載 — 加速全來自這裡。
- **Drain (排空)**: 最後 `k − 1` 步, 前段 stage 已經沒有新資料, 逐一收工。

Fill + drain 合稱 **bubble** (氣泡)、**pipeline overhead** 或 **ramp**。 Bubble 佔比 = `(k − 1) / (N + k − 1)`。 N 越大、k 越小, bubble 越可忽略。 這條式子會在第 5 章 pipeline parallelism 以 `(p − 1)/(m + p − 1)` 的形式再次出現 (p = stage 數, m = micro-batch 數)。

<Callout variant="warn" title="落到真實場景: benchmark 只跑幾次會低估吞吐">
這條 bubble 公式最常在 benchmark 咬人: 你想量一個 pipeline 的穩態吞吐, 卻只跑了 8 個 iteration、深度 k=8, 前面 fill 加後面 drain 就吃掉近一半時間 (46.7%, 見下題), 量到的 throughput 遠低於真實穩態。 這也是為什麼 profiling 一定要先跑 warmup、把 fill 段丟掉, 只量中間的 steady state。 同理, 一個只推論十幾個 token 就結束的短 request, 幾乎全程都在 pipeline 沒滿的斜坡上, 看到的 tokens/sec 不能拿來估長輸出的吞吐。
</Callout>

<Exercise id="sp1-e2" title="Bubble 佔比" kind="paper" level="進階">
某 pipeline 深度 k=8。 (a) N=8 時 bubble 佔總時間多少比例? (b) 要把 bubble 壓到 10% 以下, N 至少要多大? (c) 這對「該切多少個 micro-batch」有什麼啟示?

<Solution>
(a) bubble = `(k−1)/(N+k−1) = 7/15 ≈ 46.7%` — N 跟 k 一樣大時幾乎一半時間在暖機/排空。

(b) 要 `7/(N+7) ≤ 0.1` → `N + 7 ≥ 70` → `N ≥ 63`。

(c) 工作項 (micro-batch) 要遠多於 pipeline 深度, bubble 才划算。 但切太多 micro-batch 又會讓每項太小、per-item overhead 上升 — 這是 pipeline parallelism 調 micro-batch 數的核心 trade-off (第 5 章)。
</Solution>
</Exercise>

## 依賴與 hazard: 為什麼不能無限加深

Pipeline 的加速上限是深度 k, 但實務上有三個限制:

1. **Stage 不平衡**: 吞吐由**最慢的 stage** 決定 (木桶效應)。 若 stage B 花 2 單位、其他花 1, 穩態每 2 步才出一項, 加速被 B 卡住。
2. **依賴 (data hazard)**: 若第 j+1 項需要第 j 項的輸出, 就不能真的重疊 — 這是 GEMM double buffering 要小心的地方 (下一個 tile 的 load 不依賴當前 compute, 才能疊)。
3. **資源競爭**: 若兩個 stage 其實搶同一個硬體 (例如都用 SM), 重疊只是假象。 Pipeline 要有效, 相鄰 stage 必須用**不同**引擎。

<Callout variant="warn" title="落到真實場景: autoregressive decode 的天生依賴">
最真實的 data hazard 就是 autoregressive decode: 生第 n+1 個 token 需要第 n 個 token 當輸入, 天生序列、無法把相鄰 token 的 forward 重疊 — 這正是 decode 只能靠加大 batch (跨 request 並行) 而非加深單一序列 pipeline 來提吞吐的根因。 另一個常見版本是在 double buffering 裡不小心讓下一塊 tile 的位址依賴當前 compute 的結果 (例如 gather / indirect load), prefetch 就發不出去、重疊破功。 判準很簡單: 下一個 stage 的輸入位址能不能在當前 stage 還沒算完前就算出來, 能, 才疊得起來。
</Callout>

<Exercise id="sp1-e3" title="瓶頸 stage" kind="paper" level="進階">
一個 3-stage pipeline: load 3 單位、compute 1 單位、store 1 單位。 (a) 穩態下每幾步完成一項? (b) 吞吐由誰決定? (c) 若把 load 拆成 3 個各 1 單位的子 stage (變成 5-stage), 穩態吞吐如何改變? 有什麼代價?

<Solution>
(a) 穩態受最慢 stage (load=3) 限制, 每 **3 步**完成一項。

(b) 由 load 決定 — 加深 pipeline 也沒用, 因為瓶頸不在深度而在單一 stage 的速率。

(c) 拆成 3 個 1 單位子 stage 後, 最慢 stage 變 1 單位, 穩態變每 **1 步**一項 (3× 吞吐)。 代價: pipeline 更深 (k=5), fill/drain 更長, 且需要更多 buffer 存 in-flight 資料。 「拆瓶頸 stage 讓各 stage 平衡」是 pipeline 設計的通則。
</Solution>
</Exercise>

## GPU 世界的 pipeline 全景

同一個重疊 idea 在 GPU 軟體堆疊的每一層都出現, 只是 stage 的身分不同。 這張表也是本 track 的地圖:

| 層級 | 兩個重疊的 stage | 同步機制 | 章節 |
| --- | --- | --- | --- |
| Kernel 內 (instruction) | global→shared load ↔ MMA compute | `__syncthreads` / mbarrier | 第 2 章 |
| Host–device | H2D/D2H copy ↔ kernel compute | CUDA stream / event | 第 3 章 |
| Data input | CPU 資料載入/前處理 ↔ GPU training step | prefetch stream / worker | 第 4 章 |
| Model (多 GPU) | stage i forward ↔ stage i+1 forward | send/recv (P2P) | 第 5 章 |
| Serving | CPU 排程 ↔ GPU forward | overlap scheduler | 第 6 章 |

<Callout variant="info" title="與 GEMM track 的關係">
第 6 章 (GEMM 優化深入) 的 double buffering / cp.async 已經是 instruction-level 的 software pipelining。 這個 track 把視角拉高, 從那個 kernel 內的 producer–consumer 一路延伸到整個推論服務。 如果你還沒讀過 GEMM track 的 double buffering, 建議先看那一節再回來。
</Callout>
