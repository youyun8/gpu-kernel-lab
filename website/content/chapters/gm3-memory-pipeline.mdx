# Memory Pipeline: Vectorized Load、Double Buffering、Async Copy

Tiling (第 2 章) 把 global traffic 壓了下來, 但 kernel 還是會週期性地卡在「等 tile 從 memory 進來」。 這一章的四個技巧都在做同一件事: **讓資料的搬運與計算重疊**, 把 memory latency 藏起來。 每一個技巧獨立一節。

## Vectorized (128-bit) Loads

用 `float4` 一次搬 128 bits, 減少 load 指令數與 address 計算 overhead。 GEMM 裡它有兩個用武之地: global → shared 的搬運, 以及 shared → register 的讀取。 搭配 tile 尺寸設計, 讓每個 thread 的 load 都對齊 128-bit:

```cpp
// Global-to-shared copy, four floats at a time. Tile sizes are chosen so each
// thread's slice is 16-byte aligned; the K dimension must be divisible by 4.
const float4* src = reinterpret_cast<const float4*>(&A[row * k + t]);
float4 v = src[threadIdx.x];
sA[threadIdx.y][threadIdx.x * 4 + 0] = v.x;
sA[threadIdx.y][threadIdx.x * 4 + 1] = v.y;
sA[threadIdx.y][threadIdx.x * 4 + 2] = v.z;
sA[threadIdx.y][threadIdx.x * 4 + 3] = v.w;
```

一次 128-bit transaction 取代四次 32-bit, memory 子系統的請求數少了 4 倍, 對已經 coalesced 的存取尤其有效。 代價是對齊要求變嚴: base pointer 與 K 都要能被 4 整除, 尾端 (第 6 章) 得特別處理。

<Callout variant="warn" title="落到真實場景: K 不是 4 的倍數, float4 直接 misaligned">
`float4` 向量化在方正的 shape (`K=4096`) 上很甜, 但真實模型常有不對齊的維度: 某個 projection 的 `K=2050`、或 vocab 相關的 GEMM `K=50257`。 一旦 base pointer 或 leading dimension 不是 16-byte 對齊, `reinterpret_cast<const float4*>` 的載入就是 `misaligned address` — 在 CUDA 上直接 kernel 崩掉、在 HIP 上可能讀到垃圾。 production kernel 的做法是: 主體走對齊的 float4 fast path, 對尾端不足 4 個的殘量 (第 6 章) 退回逐一 scalar load。 別假設輸入永遠對齊 — 那是 serving 時最常見的 crash 來源之一。
</Callout>

## Double Buffering — 把 load 藏進 compute

單 buffer 的主迴圈是「load tile → sync → compute → sync」, load 期間算術單元閒著。 Double buffering 配置兩組 shared memory buffer: 算 buffer 0 的同時, 預取下一個 k-tile 進 buffer 1, 每輪交換 — 這就是 software pipelining。

<DoubleBufferingFigure />

```cpp
// Two shared-memory stages: compute on `cur` while prefetching into `nxt`.
__shared__ float sA[2][kTile][kTile];
__shared__ float sB[2][kTile][kTile];
int cur = 0;
loadTile(sA[cur], sB[cur], /*t=*/0);
__syncthreads();
for (int t = kTile; t < k; t += kTile) {
    int nxt = cur ^ 1;
    loadTile(sA[nxt], sB[nxt], t);   // issue loads for the next tile
    computeTile(sA[cur], sB[cur]);   // FMAs overlap the loads in flight
    __syncthreads();
    cur = nxt;
}
computeTile(sA[cur], sB[cur]);       // epilogue: last tile
```

理想情況每個 k-tile 的成本從 `L + C` 變成 `max(L, C)` — 當 compute 足以覆蓋 load, memory latency 就被完全藏掉。

<Callout variant="insight" title="落到真實場景: profiler 上 long-scoreboard stall 掉一半">
double buffering 的效果在 `ncu` 上是看得見的具體數字: 單 buffer 版本的 warp state 會被 `stall_long_scoreboard` (等 global load 回來) 主導, 常佔總 stall 的 40–60%。 加上 double buffering 後, 因為下一個 k-tile 的 load 在算當前 tile 時就已發出去, 這個 stall 通常砍到剩一半以下, GFLOP/s 隨之往上跳。 這也是為什麼調 GEMM 時第一個要盯的 metric 就是 long-scoreboard 佔比 — 它直接告訴你 memory latency 有沒有被 compute 蓋住。
</Callout>

## Multi-stage Async Copy — cp.async 與 TMA

傳統的 global → shared 搬運會先把資料經過 register 中轉 (load 到 register, 再 store 到 shared)。 這佔用 register、也讓 load 阻塞 warp。 Ampere 引入的 `cp.async` 讓資料**直接 DRAM → shared, 繞過 register**, 且是非同步的 — issue 出去後 warp 可以繼續算, 到需要時才 `cp.async.wait`。

```cpp
// Ampere+ cp.async: DRAM -> shared without staging through registers.
// Kicks off N stages of prefetch, then waits one stage behind the compute.
#include <cuda_pipeline.h>
__pipeline_memcpy_async(&sA[stage][ty][tx], &A[row * k + t], sizeof(float4));
__pipeline_commit();
// ... later, before consuming `stage`:
__pipeline_wait_prior(kStages - 1);
__syncthreads();
```

因為 prefetch 不再佔用 register, pipeline 可以疊到 **3–5 個 stage** (不只 double)。 Hopper 的 **TMA** (Tensor Memory Accelerator) 更進一步, 用一個 descriptor 描述整塊 tile 的 copy, 連 address 計算都交給硬體。 AMD 側對應的機制是 LDS 的 direct-to-LDS 載入路徑與 `buffer_load` 系列指令。

<Callout variant="warn" title="stage 數不是越多越好">
每多一個 stage 就多一份 shared memory buffer, 直接吃掉 occupancy。 3-stage 通常是甜蜜點; 要疊更多得先確認 shared memory 容量與 register 壓力還有餘裕。 這是第 7 章 autotuning 的一個重要旋鈕。
</Callout>

## Swizzle — 讓 pipeline 不被 bank conflict 拖住

把 tile 搬進 shared memory 後, thread (或 MMA 指令) 讀回來時若多個 lane 撞到同一個 bank, 就 serialize 了。 最簡單的解法是 **padding** (每列多補一個 element 讓 stride 錯開); 更進階、且 MMA 必需的是 **swizzle** — 用 XOR 之類的位元重排把 element 打散到不同 bank:

```cpp
// XOR swizzle: permute the column so threads of a warp hit distinct banks.
// Common in MMA pipelines where the fragment layout is fixed by hardware.
int swizzledCol = col ^ ((row & 7) << 2);
sA[row][swizzledCol] = value;
```

Padding 簡單但浪費 shared memory 且對 128-bit 存取不一定夠; swizzle 不浪費容量, 但 index 計算較複雜 — 這也是第 4 章手寫 MMA kernel 困難、CUTLASS/CK 有價值的原因之一。

<Callout variant="warn" title="落到真實場景: 32-way bank conflict 把 shared 讀慢 32 倍">
把 half tile 以 naive row-major 塞進 shared memory, 再用 `ldmatrix` 餵 Tensor Core, 一個 warp 的 32 個 lane 很可能全撞進同一組 bank — shared memory 讀取被 serialize 成 32 次, `ncu` 的 `l1tex__data_bank_conflicts_pipe_lsu_mem_shared` 會爆高, mainloop 明明該是 compute-bound 卻卡在 shared load。 padding (每列多補幾個 element 把 stride 錯開) 是最快的止血, 但真正對齊 MMA fragment 佈局的是 XOR swizzle。 這正是手寫 MMA kernel 最惱人的一段, 也是 CUTLASS 的 CuTe 用 layout 代數幫你把 swizzle 算對的價值所在。
</Callout>
