# 認識 Kernel Library 生態

讀到這裡你已經能手寫相當快的 GEMM。 但殘酷的事實是: 你花一個月寫的 kernel, cuBLAS 一行呼叫就打平甚至超越。 那學這些還有意義嗎? 有 — 但你得知道什麼時候該收手、把工作交給 library。 這一章畫出生態地圖, 幫你做這個決定; Track E 會對地圖上的每個重要節點深入展開。

## 三種抽象層次

**CUTLASS (NVIDIA)** 是一套 C++ template library, 把 GEMM/conv 拆成分層的可組合元件: device → kernel → threadblock → warp → instruction (MMA)。 你可以在任一層插入自訂 epilogue、fusion、data type, 同時複用底層高度優化的 pipeline。 它是「我要 library 級效能, 但需要客製」的答案。 AMD 的對應是 **Composable Kernel (CK)**, 同樣走 template 分層、可組合的路線。 第 22、23 章分別深入兩者。

**Triton** 是一個 Python DSL: 你用類似 NumPy 的 block-level 語法寫 kernel, compiler 幫你處理 shared memory 配置、coalescing、pipelining。 生產力極高, 可攜性好 (同一份 Triton 可跑 NVIDIA 與 AMD), 峰值效能通常略低於手調 CUTLASS 但常常「夠好」。 它也是 `torch.compile` 的 codegen backend (第 20 章)。

**cuBLAS / hipBLASLt** 是黑盒 BLAS: 最省事、對標準 GEMM 通常最快, 但無法客製 fusion (cuBLASLt 的 epilogue 選單除外)、也看不到內部。 第 13 章講過它們內部的 kernel zoo 與 heuristic。 呼叫端長這樣 — 注意 BLAS 有自己的 handle 與 status code, 錯誤檢查和 runtime API 是兩套:

```cpp
// cuBLAS: one call replaces the whole sgemm_step0..step4 journey.
cublasHandle_t handle;
BLAS_CHECK(cublasCreate(&handle));
const float alpha = 1.0f, beta = 0.0f;
BLAS_CHECK(cublasSgemm(handle, CUBLAS_OP_N, CUBLAS_OP_N, m, n, k,
    &alpha, dev_a, m, dev_b, k, &beta, dev_c, m));

// hipBLAS mirrors the API one-to-one:
// hipblasSgemm(handle, HIPBLAS_OP_N, HIPBLAS_OP_N, m, n, k, ...)
```

兩個容易踩的細節: BLAS 介面是 **column-major** (Fortran 傳統, 所以 row-major 的 C 陣列常用「交換 A/B 順序」的技巧), 而且 `cublasSgemm` 是非同步的 — 計時要用 event 或先 synchronize (第 4 章)。

| 需求 | 建議 |
| --- | --- |
| 標準 GEMM, 要最快、最省事 | cuBLAS / hipBLASLt |
| 需要客製 epilogue / fusion / 非標準 dtype, 且要接近 peak | CUTLASS / Composable Kernel |
| 快速原型、可攜性優先、團隊用 Python | Triton |
| 學習、教學、極端客製的研究 | 手寫 kernel |

<Callout variant="insight" title="落到真實場景: 為一個 fp8 attention 選層次">
假設你在做 LLM 推論, 要上 fp8 的 fused QKV projection + bias。 先問這張表: shape 固定嗎? serving 的 hidden size 通常固定, 所以 autotune 划算。 需要 fusion 嗎? 要把 bias 與 dequant 塞進 epilogue, 黑盒 `cublasSgemm` 立刻出局。 於是路徑很清楚: NVIDIA 上用 CUTLASS 的 epilogue visitor 把 bias/scale 融進 GEMM 收尾, ROCm 上先試 AITER/CK; 但若你只是想兩週內先跑起來、驗證數值正確, 先寫個 Triton 版拿到 correctness 與收益上限, 再決定要不要下探到 CUTLASS。 這就是「地圖」的實際用法 — 不是背 API, 而是用需求把選項砍到剩一兩個。
</Callout>

## 這張地圖如何接到真實世界

你平常用的框架, 底下全是這張地圖上的元件:

- **PyTorch**: `torch.mm` 走 cuBLAS/hipBLASLt;`scaled_dot_product_attention` 走 FlashAttention/內建 kernel; fp8 的 `_scaled_mm` 用 CUTLASS;`torch.compile` 生成 Triton。
- **vLLM / SGLang** (LLM serving): attention 用 FlashAttention/FlashInfer/自製 PagedAttention kernel, GEMM 用 cuBLASLt/CUTLASS/hipBLASLt, MoE 與量化 kernel 用 CUTLASS/Triton/AITER (AMD)。
- **TensorRT-LLM**: NVIDIA 的推論引擎, 內部大量使用 CUTLASS 生成的 kernel 與手寫 attention。

換句話說: 學會讀懂這些 library, 你就能讀懂 (並優化) 現代 AI infra 的底層。 Track E (第 22–25 章) 逐一拆解。

<Callout variant="insight" title="落到真實場景: 從 profiler trace 反查是哪個 library">
你 profiling 一個 Llama 推論的 decode step, Nsight Systems 的 timeline 上出現一個叫 `void cutlass::Kernel2<...>` 的熱點、旁邊是 `flash_fwd_kernel`, 再往下是幾個 `triton_poi_fused_*`。 這串名字直接告訴你底層是誰: `cutlass::` 前綴 = fp8/量化 GEMM 走 CUTLASS, `flash_` = attention 走 FlashAttention, `triton_..._fused_` = `torch.compile` 生成的 elementwise fusion。 認得出這些前綴, 你才知道某個熱點該去哪個 repo 找可調參數 (CUTLASS 換 tile config、Triton 看 Inductor 有沒有 fuse 好), 而不是盲目重寫。
</Callout>

## 什麼時候該自己寫, 什麼時候該用 Library

<Callout variant="insight" title="決策原則">
先量測: 如果 cuBLAS/hipBLASLt 已經佔了你 kernel 時間的大宗且接近 roofline, 別重寫, 把精力放在別處。 當你需要的是「library 不提供的 fusion」或「奇特的 shape/dtype」, 才輪到 CUTLASS/CK 或手寫。 手寫 from scratch 的價值主要在學習與研究, 以及 library 完全沒覆蓋的場景。
</Callout>

一個實用的決策順序 (由省力到費力): 內建 op → `torch.compile` → library 的 fusion 選項 (cuBLASLt epilogue) → Triton → CUTLASS/CK → 手寫。 每往下一層, 開發與維護成本增加一個量級, 只有量測證明上一層不夠時才往下走。

## Autotune 的角色

同一個 GEMM 在不同 shape、不同 GPU 上的最佳 tile size / stage 數 / swizzle 都不同。 Library (cuBLASLt、CK、Triton 的 autotuner) 會在候選 config 中搜尋最佳解。 你自己寫的 kernel 若要對多種 shape 都快, 也得建立類似的 autotune 機制 — 這通常比寫單一 kernel 更費工, 也是 library 真正的價值所在。

Autotune 的成本模型也值得理解: Triton 的 `@triton.autotune` 在首次遇到新 shape 時實測所有候選 config (冷啟動慢), 之後快取命中; cuBLAS 的 heuristic 零冷啟動但可能次優; TunableOp/CUTLASS profiler 走離線搜索。 選哪種, 取決於你的 shape 分佈是否固定、能否接受首次延遲。

<Callout variant="warn" title="落到真實場景: autotune 在動態 shape 下反覆冷啟動">
一個常見翻車: 你在 LLM serving 開了 `torch.compile` 或 Triton autotune, 但 prefill 的 sequence length 每個 request 都不同 (37、512、1024...)。 如果 autotune 的 `key` 直接綁 seqlen, 每個新長度都觸發一次完整 config 搜索, 首個 request 的 latency 突然噴到數百 ms — 使用者看到的是「偶發卡頓」。 修法是把 shape bucket 化 (padding 到 128 的倍數再進 kernel), 或改用 PyTorch 的 `TORCH_CUDABLAS_TUNABLEOP` 做離線 tuning 把結果存檔, 上線時只讀不搜。 這也是為什麼 production 常寧可用 heuristic 型的 cuBLASLt: 犧牲幾 % peak 換掉尾端延遲的不可預測性。
</Callout>

## Library Selection Playbook

做選型時把需求寫成矩陣:

- **Shape stability**: 固定 shape 適合離線 autotune; 高度動態 shape 偏向 heuristic 或 persistent scheduler。
- **Fusion need**: 只有 bias/ReLU/GELU 時先試 cuBLASLt/hipBLASLt epilogue; 任意 epilogue 才進 CUTLASS/CK/Triton。
- **Portability**: 同一 codebase 要跑 CUDA/ROCm, Triton 或 ATen native op 的維護成本最低。
- **Peak requirement**: 要最後 5% 且 shape 固定, CUTLASS/CK + profiler 值得; 否則 Triton 常更划算。
- **Build constraints**: C++ template library 會拉長 build time; Python JIT 會有 cold start; 黑盒 BLAS 部署最簡單。
- **Debuggability**: 手寫/Triton 最容易插入檢查; 黑盒最快但可觀測性最低。

例子: 一個固定 shape 的 fp8 GEMM + custom dequant + activation, NVIDIA 上可用 CUTLASS epilogue visitor; ROCm 上先試 AITER/CK; 若兩邊都要快速交付, Triton prototype 先建立 correctness 和收益上限, 再決定是否下探。

<Callout variant="insight" title="落到真實場景: 用 scorecard 擋掉一個「聽起來很酷」的重寫">
團隊有人提議把某個 MoE 的 gating GEMM 用 CUTLASS 手刻。 套 playbook: portability — 這個服務同時上 H100 和 MI300, 手刻 CUTLASS 意味著要另外維護一份 CK 版本, 維護成本翻倍。 build constraints — CUTLASS template 會讓 CI 的編譯時間從 5 分鐘拉到 20 分鐘。 peak requirement — profiler 顯示這個 GEMM 只佔 end-to-end 的 6%, 就算榨到 peak 整體也才快不到 3%。 結論寫進 review: 先用 Triton 版跨兩平台交付, CUTLASS 留到這個 op 真的變成 top-3 熱點再說。 Scorecard 的價值不是選最快的, 而是把「值不值得」量化成能寫進 PR 描述的理由。
</Callout>

`blas_baseline` 用 cuBLAS (CUDA) 或 hipBLAS (HIP) 跑同一個 2048³ SGEMM (CMake 找不到 BLAS library 時會自動略過這個 target)。 把它的 GFLOP/s 和你手寫的 `sgemm_step4_double_buffer` 並排比較, 誠實面對差距 — 然後想想: 剩下的差距是來自 autotune、還是某個你還沒做的 pipeline 技巧?
</LabBox>

<Solution>
預期結果與判讀: fp32 SGEMM 上, 手寫 step4 通常能到 cuBLAS 的 60–85% — 剩下的差距主要來自: (1) library 對這個形狀挑了更優的 tile/stage 組合 (autotune); (2) 更深的 software pipeline 與更精細的指令排程; (3) 部分版本走 tf32 Tensor Core 路徑 (若你沒關 math mode, 這不是同一場比賽 — 用 `CUBLAS_PEDANTIC_MATH` 對齊)。 行動建議: 若差距 < 20%, 對大多數用途手寫版已「夠好」; 若你需要那 20%,第 22 章的 CUTLASS 是比繼續手調更經濟的路。

完整程式: [blas_baseline.cpp](https://github.com/youyun8/gpu-kernel-lab/blob/main/kernels/03-gemm/blas_baseline.cpp)。
</Solution>
