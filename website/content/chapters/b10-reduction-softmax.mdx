# Reduction 與 Softmax 優化系列

Reduction 看起來是最簡單的 kernel — 把一堆數字加起來 — 卻是最能展現 GPU 優化層次的題目。 從最天真的版本到接近頻寬上限, 中間有五、六個明確的台階。 學會 reduction, 你就有了 softmax、layernorm、attention 的地基。

## Reduction 的優化階梯

**Naive:** 每個 thread `atomicAdd` 到一個全域累加器 — 嚴重 contention, 慢到不行。

**Tree reduction in shared memory:** block 內用 shared memory 做二元樹相加。 要小心兩件事: early 版本的 `if (tid % (2*s) == 0)` 造成 divergence 與 bank conflict; 改成 `for (s = blockDim/2; s > 0; s >>= 1)` 的 sequential addressing 版本可同時解決兩者。

**Warp shuffle:** 最後 32/64 個元素不必再進 shared memory, 直接用第 9 章的 `warpReduceSum` 收尾, 省掉多輪 `__syncthreads()`。

**Vectorized loads + grid-stride:** 每個 thread 用 `float4` 一次讀 4 個、並以 grid-stride loop 處理多筆, 提高每 thread 的工作量與 memory 效率。

```cpp
// Block reduction: shared-memory tree down to one warp, then warp shuffle.
__global__ void reduceSum(const float* in, float* out, int n) {
    __shared__ float smem[256];
    int tid = threadIdx.x;
    float sum = 0.0f;
    // grid-stride loop with vectorized-friendly accumulation
    for (int i = blockIdx.x * blockDim.x + tid; i < n; i += blockDim.x * gridDim.x) {
        sum += in[i];
    }
    smem[tid] = sum;
    __syncthreads();
    for (int s = blockDim.x / 2; s > warpSize; s >>= 1) {
        if (tid < s) smem[tid] += smem[tid + s];
        __syncthreads();
    }
    if (tid < warpSize) {
        float v = smem[tid];
        for (int o = warpSize / 2; o > 0; o >>= 1) v += __shfl_down_sync(0xffffffff, v, o);
        if (tid == 0) out[blockIdx.x] = v;
    }
}
```

<Callout variant="insight" title="落到真實場景: 為什麼 reduction 的目標是「讀一遍的速度」">
量一支 sum / norm reduction 時, 別拿 FLOP 當指標 — 它幾乎不算數, 你唯一要追的是 achieved bandwidth 對 HBM peak 的百分比。 上面這支 kernel 的理論極限就是「把輸入讀一遍」的時間 (輸出只有 gridDim 個 partial)。 naive 的 atomic 版可能只跑到峰值頻寬的 5–10%, 這支 shared-tree + warp-shuffle 版則應該逼近 80–90%。 所以驗收標準很明確: 用第 4 章的 benchmark 算出 GB/s, 除以 spec 上的 HBM 頻寬, 接近 1 就沒有再優化的空間了。
</Callout>

## 最後一哩: 跨 block 的合併

每個 block 產出一個 partial sum 之後, 還要合併成一個值。 三種做法各有取捨:

- **兩段式 kernel**: 第一個 kernel 產出 `gridDim.x` 個 partial, 第二個 kernel (單 block) 收尾。 簡單、確定性強, 是預設選擇。
- **`atomicAdd` 收尾**: 每個 block 的 thread 0 直接 `atomicAdd(out, v)`。 省一次 launch, block 數不多時 contention 可忽略; 代價是浮點加法順序不定, 結果在 run 之間可能有 bit-level 差異 (對測試的容忍度設計有影響)。
- **單 kernel + grid 同步**: 用 cooperative launch 的 grid-wide barrier。 限制多 (grid 必須全駐留), 特定場景才值得。

<Callout variant="warn" title="浮點 reduction 的順序問題">
浮點加法不符合結合律, 不同的 reduce 順序給出略不同的結果。 GPU reduction 的求和順序本來就和 CPU 逐一累加不同, 所以 correctness check 一定要用相對誤差容忍, 不能 bit-exact 比對; 而 atomic 版本連 run 對 run 都不保證一致 — 需要「可重現」時選兩段式。
</Callout>

## Softmax 與 numerical stability

Softmax 需要兩個 reduction: max 與 sum-of-exp。 直接算 $e^{x_i}$ 會 overflow, 所以要先減掉 max:

$$
\text{softmax}(x_i) = \frac{e^{x_i - \max_j x_j}}{\sum_k e^{x_k - \max_j x_j}}
$$

實作上就是把本章的 reduction 用兩次 (max 一次、sum 一次), 再做一次 elementwise 正規化 — naive 版本共掃三遍資料。 Row 夠短時一個 warp/block 處理一個 row, 中間值全程留在 register, 這正是 fused softmax kernel 的形狀 (第 20 章用 Triton 重寫的就是它)。

<Callout variant="insight" title="落到真實場景: 不減 max 的 softmax 在 logits 上直接變 NaN">
LLM 最後一層對著幾萬個 vocab 算 softmax, logits 動輒到 30~50; `expf(50)` 已經是 5e21, 再大幾個就 overflow 成 inf, 接著 inf / inf = NaN, 整個 loss 爆掉。 這不是理論顧慮 — 混合精度訓練裡漏掉 max-subtraction 是很典型的 NaN 來源。 本章的三遍作法 (max → sum → normalize) 與 online 版都內建這個減 max, 所以數值穩; 自己手寫 fused softmax 時, max-subtraction 是不能省的第一步, 而不是「算完 exp 再回頭補救 overflow」。
</Callout>

## Online softmax: 一次掃描搞定

Naive softmax 掃三遍 (找 max、算分母、正規化)。 **Online softmax** 只掃一遍: 邊走邊維護 running max 與 running sum, 遇到更大的 max 就用修正因子 rescale 之前的 sum。 這正是 FlashAttention 的核心技巧 — 讓 softmax 不必物化整個 row。

$$
m_{new} = \max (m_{old}, x_i), \quad d_{new} = d_{old} \cdot e^{m_{old} - m_{new}} + e^{x_i - m_{new}}
$$

```cpp
// Single-pass (online) softmax normalizer: one read per element.
float m = -INFINITY, d = 0.0f;
for (int i = 0; i < n; ++i) {
    float x = row[i];
    float mNew = fmaxf(m, x);
    d = d * expf(m - mNew) + expf(x - mNew); // rescale old sum, add new term
    m = mNew;
}
// softmax(x_i) = exp(row[i] - m) / d  (one more pass, or fuse into consumer)
```

兩個 (m, d) pair 還可以用同樣的 rescale 規則合併 — 這讓 online softmax 能平行化: 每個 thread/warp 各自維護局部的 (m, d), 最後樹狀合併, 與本章的 reduction 階梯完全同構。

<Callout variant="insight" title="為什麼 online softmax 重要">
它把 softmax 的 memory 需求從「整個 row」降到「幾個 scalar」, 讓 attention 可以 tile、可以 fuse、可以省下對長序列中間結果的往返。 理解這一步, 你就摸到了現代 attention kernel 的門 — FlashAttention (第 24 章) 就是 online softmax + tiling 的組合。
</Callout>

## 更多 Reduction Variants

Reduction 家族很大, 實務上會依 shape 選不同版本:

- **Warp-per-row**: row 長度 `<= warpSize` 時, 一個 warp 處理一列, 用 shuffle all-reduce, 適合小 softmax/layernorm。
- **Block-per-row**: row 較長時, 一個 block 處理一列, 每 thread 多元素 grid-stride, shared memory 合併 warp partials。
- **Multi-block per row**: row 很長時, 多 blocks 處理同一列並輸出 partial, 第二階段再 reduce。
- **Persistent reduction**: 大量短 row 且長度不均時, persistent scheduler 動態取 row, 減少 tail effect。
- **Segmented reduction**: key/value 已排序或有 segment offsets 時, 每段各自 reduce。 Histogram、embedding gradient、sparse ops 常用。

數值上也要選策略: 普通 sum 可用 tree reduction; 長向量或精度敏感可用 pairwise/Kahan 類補償, 但會增加 FLOP 和 register。 Softmax 則應優先用 max-subtraction 或 online normalizer, 不要靠 `expf` 之後再補救 overflow。

<Callout variant="insight" title="落到真實場景: 用 shape 決定 reduction 變體">
同樣是「對最後一維求和 / 求 max」, 真實 workload 的形狀決定你該挑哪個變體。 BERT 類的 LayerNorm, hidden=768/1024 的短 row → warp-per-row (一個 warp 一列, shuffle all-reduce) 最省。 GPT 對 vocab=50k 算 softmax 的長 row → block-per-row 甚至 multi-block per row + 第二階段 reduce。 推薦系統的 embedding 依 segment offset 做 segmented reduction。 而一批長度參差不齊的 sequence, 用 persistent kernel 動態領 row 才能壓掉 tail effect。 選錯變體 (例如拿 block-per-row 去做幾百萬條超短 row) 會讓一半的 thread 閒著。
</Callout>
