# Memory Coalescing

為什麼同一個 vector add, 只是把索引從 `i` 改成 `i * 2`, achieved bandwidth 就掉了一半? 程式邏輯幾乎沒變, 搬的資料量看起來也一樣。 答案藏在 global memory 的存取粒度裡: GPU 不是一次搬一個 float, 而是一次搬一整段 memory。

<Callout variant="insight" title="先給直覺">
把 global memory 想成一箱一箱出貨的倉庫, 每箱固定 128 bytes。 你要的東西如果剛好塞在同一箱, 一次就拿完; 如果分散在 32 箱, 你得叫 32 台車, 而且每台車大部分空間都載空氣。 Memory coalescing 就是「讓一個 warp 的 32 個 threads 盡量只叫最少的車」。
</Callout>

## 硬體怎麼看一個 warp 的存取

當一個 warp (NVIDIA 32 threads, AMD wavefront 64 threads) 執行一條 load 指令時, memory subsystem 會收集這 32/64 個 threads 想要的位址, 然後把它們切成固定大小的 memory transactions (常見是 32 或 128 bytes 的 segment)。 關鍵指標很簡單:

$$
\text{efficiency} = \frac{\text{useful bytes requested}}{\text{bytes actually moved}}
$$

如果一個 warp 的 32 個 threads 存取連續且對齊的 32 個 float (128 bytes), 硬體只要一次 transaction 就能滿足, efficiency = 100%。 一旦 threads 之間有 stride, touched segments 變多, 搬進來卻沒被用到的 bytes 就是白白浪費的 bandwidth。

把 stride 從 1 拉到 2, 你會看到 transaction 數量翻倍、efficiency 掉到 50%。 這正是 AoS (array of structs) 常見的陷阱。

## AoS vs. SoA: 同一份資料, 兩種下場

假設每個 particle 有 `x, y, z` 三個 float, 你要對所有 particle 的 `x` 做運算。

<PlatformTabs>
<Platform name="CUDA">

```cpp
// AoS: x is interleaved with y, z. A warp reading .x has stride 3 -> uncoalesced.
struct Particle { float x, y, z; };
__global__ void scaleXAos(Particle* p, float k, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) p[i].x *= k; // threads touch bytes 0, 12, 24, ... : 3x traffic
}

// SoA: all x values are contiguous. A warp reads 32 consecutive floats -> coalesced.
__global__ void scaleXSoa(float* x, float k, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) x[i] *= k; // threads touch bytes 0, 4, 8, ... : 1 transaction
}
```

</Platform>
<Platform name="HIP / ROCm">

```cpp
// Identical logic under HIP; a wavefront is 64 lanes on CDNA GPUs.
struct Particle { float x, y, z; };
__global__ void scaleXAos(Particle* p, float k, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) p[i].x *= k; // strided access, poor global efficiency
}

__global__ void scaleXSoa(float* x, float k, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) x[i] *= k; // contiguous, coalesced
}
```

</Platform>
</PlatformTabs>

SoA 讓「同一個欄位跨 element」變成連續存取, 這正是 warp 想要的排列方式。 改資料結構往往比改 kernel 邏輯更有效。

折衷方案也存在: 如果 kernel 總是同時用到 `x, y, z`, 那 AoS 反而只要一次寬存取; 或用 AoSoA (把 SoA 以 warp 大小分塊交錯) 兼顧兩種模式。 原則是: **依照「warp 在同一個 cycle 想要什麼」來排資料**, 而不是依照物件導向的直覺。

## Misaligned access 的代價

就算 stride = 1, 如果起始位址沒對齊 segment 邊界, 一個 warp 的存取會跨兩個 segment, 需要 2 次 transaction 而不是 1 次。 對 memory-bound kernel 來說, 這等於憑空多付近一倍的 bandwidth。 實務上: 讓 allocation 對齊、避免對 buffer 做奇數 byte 的 offset、必要時用 padding 把每 row 對齊。

`cudaMalloc` / `hipMalloc` 回傳的指標本身保證至少 256-byte 對齊; misalignment 幾乎都來自你自己的 offset 運算 (例如 `ptr + 3`、對 2D 陣列用未 padding 的 row width)。 對 2D 資料用 `cudaMallocPitch` 或自行把 row stride 補到 128 bytes 的倍數, 可以讓每一 row 的開頭都對齊。

## 讀和寫都要 coalesced

Coalescing 的規則對 load 與 store 同樣成立, 但兩者的病徵不同: uncoalesced read 會多搬進沒用的 bytes; uncoalesced write 更糟, 部分寫入一個 segment 可能觸發 read-modify-write。 經典案例是 matrix transpose — 讀 row-major 是 coalesced 的, 寫 column-major 天生是 strided 的, 這正是第 7 章要用 shared memory 解決的問題。

在 profiler 裡驗證: Nsight Compute 看 `l1tex__t_sectors_pipe_lsu_mem_global_op_ld` 相對理想值的比例 (或 Memory Workload Analysis 的 "sectors per request"); omniperf 看 memory chart 的 coalescing / L1-L2 request 放大倍率。 理想的 fp32 連續存取, 每個 warp request 對應 4 個 32-byte sector; 數字更高就是 access pattern 有問題。

<Callout variant="warn" title="常見誤解">
「反正資料都要搬, coalesced 與否只差常數。」錯。 對 bandwidth-bound kernel, efficiency 直接乘在你的有效頻寬上: efficiency 50% 代表你的 kernel 上限就只有硬體頻寬的一半, 再怎麼調 occupancy 也追不回來。
</Callout>

## Cache 能救回多少?

L1/L2 會緩解部分懲罰 — 第一個 warp 搬進來的 segment, 相鄰 warp 若剛好要用就是 cache hit。 所以「stride 小」的 uncoalesced 存取實測可能沒有理論那麼慘。 但別把這當藉口: cache 容量有限, 資料量一大 (超過 L2) 或存取分散, 懲罰就會全額浮現。 用大 buffer 量測才反映真實情況 — 這也是 lab 的 `stride_bandwidth` 用大陣列的原因。

## Advanced Access Patterns

以下六個場景涵蓋了實務上絕大多數「access pattern 不理想」的處理方式。 每一個都有對應的可編譯 benchmark 在 `kernels/02-memory/access-patterns/`。

### AoSoA: 分塊交錯的折衷

把資料切成 warp/wavefront 大小的小 SoA chunk, chunk 之間仍保留 object grouping。 粒子模擬和 graph feature 常用這個折衷 — 讀單一欄位時 warp 內連續 (像 SoA), 同一物件的所有欄位又在同一個 chunk 附近 (像 AoS)。

```cpp
// AoSoA: each chunk stores 32 x's, then 32 y's, then 32 z's.
struct ParticleChunk {
    float x[32];
    float y[32];
    float z[32];
};

// Lane i of a warp reads x[i] within one chunk -> one contiguous
// 128-byte segment, coalesced exactly like SoA.
__global__ void scaleXAosoa(ParticleChunk* chunks, float k, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) chunks[i / 32].x[i % 32] *= k;
}
```

`aosoa_layout` benchmark 用同一個「scale x」操作跑 AoS/SoA/AoSoA 三種佈局: SoA 與 AoSoA 頻寬相近, AoS 因為 stride-3 大約掉三倍。

### Vectorized loads 與 tail cleanup

連續、對齊且每 thread 有多元素工作時, 用 `float4` / `int4` 把每條指令搬的資料從 4 bytes 提高到 16 bytes, 降低指令數與 index 運算。 長度不是 4 的倍數時, 主體用 float4、尾端 (n % 4 個元素) 用 scalar cleanup kernel 收尾:

```cpp
// Main body: each thread handles one float4 (16 bytes).
__global__ void saxpyVec4Body(const float4* x, float4* y, float a, int n4) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n4) {
    float4 xv = x[i], yv = y[i];
    yv.x = a * xv.x + yv.x;  yv.y = a * xv.y + yv.y;
    yv.z = a * xv.z + yv.z;  yv.w = a * xv.w + yv.w;
    y[i] = yv;
    }
}

// Tail cleanup: the last n % 4 elements, one tiny scalar launch.
__global__ void saxpyTail(const float* x, float* y, float a, int tail_start, int n) {
    int i = tail_start + threadIdx.x;
    if (i < n) y[i] = a * x[i] + y[i];
}
```

注意前提: `reinterpret_cast<float4*>` 要求 16-byte 對齊, device allocation 本身沒問題, 但對指標做過奇數 offset 就會 crash。 `vectorized_tail` 用 n = 2²⁴ + 3 驗證主體 + 尾端的正確性。

### Cooperative loading: block 協力搬 tile

一個 block 協力把 global 資料整理成 shared memory 中的規則 tile, 之後的 compute 階段就能 coalesced 或 conflict-free。 關鍵是 load phase 與 compute phase 之間的 `__syncthreads()` — 沒有它, 有些 thread 會讀到還沒被寫入的 tile 元素:

```cpp
__global__ void boxFilterShared(const float* in, float* out, int n) {
    __shared__ float tile[kBlockSize + 2 * kRadius];
    int i = blockIdx.x * blockDim.x + threadIdx.x;

    // Phase 1: everyone loads its own element (coalesced).
    tile[kRadius + threadIdx.x] = (i < n) ? in[i] : 0.0f;
    // Phase 2: first kRadius threads fetch the left/right halos.
    if (threadIdx.x < kRadius) { /* load halo elements */ }

    __syncthreads();  // no one reads tile[] before everyone wrote it

    float sum = 0.0f;
    for (int r = -kRadius; r <= kRadius; ++r)
    sum += tile[kRadius + threadIdx.x + r];
    out[i] = sum / (2 * kRadius + 1);
}
```

`cooperative_tile_load` 拿它對比「每個 thread 自己發 17 個 global load」的 naive 版: 邏輯結果相同, 但 load 指令數差 17 倍。

### Gather vs. scatter

如果只能讓讀或寫其中一側 coalesced, 優先讓資料量較大、重用較低的一側 coalesced。 Gather (`out[i] = in[idx[i]]`) 天生寫入連續; scatter (`out[idx[i]] = in[i]`) 寫入分散, 若目標還會碰撞就必須用 atomic, 而 atomic 對同一位址會序列化。 下面的互動元件比較兩者, 並展示 aggregation 怎麼減少 atomic 數量:

<GatherScatterVisualizer />

Scatter + atomic 的標準解法是先在 block 內聚合, 再一次寫出:

```cpp
// Block-aggregated scatter: shared-memory histogram absorbs
// collisions cheaply, then one flush issues kNumBins global
// atomics per block instead of kBlockSize.
__global__ void scatterAtomicAggregated(const int* idx, unsigned int* bins, int n) {
    __shared__ unsigned int local_bins[kNumBins];
    for (int b = threadIdx.x; b < kNumBins; b += blockDim.x) local_bins[b] = 0u;
    __syncthreads();

    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) atomicAdd(&local_bins[idx[i]], 1u);
    __syncthreads();

    for (int b = threadIdx.x; b < kNumBins; b += blockDim.x)
    if (local_bins[b] != 0u) atomicAdd(&bins[b], local_bins[b]);
}
```

`gather_scatter` 對 4M 筆資料實測 gather、naive scatter、block-aggregated scatter 三種路徑的差距。 Atomic contention 的更多變化 (warp aggregation、privatization) 見第 26 章。

### Row padding: 對齊每一 row 的開頭

2D tensor 的 leading dimension 補到 128 bytes 的倍數, 避免每 row 開頭逐漸漂移造成 misalignment。 例如 1000 個 float 的 row 是 4000 bytes — 每往下一 row, 開頭就在 128-byte segment 內漂移 32 bytes, 大部分 row 的第一個 warp 都要多一次 transaction:

```cpp
// ld = 1000: row r starts at byte 4000*r, misaligned for most r.
// ld = 1024: row r starts at byte 4096*r, always 128-byte aligned,
//            at the cost of 2.4% wasted memory.
__global__ void scaleRows(float* data, int ld, int rows, int cols, float k) {
    float* row = data + static_cast<size_t>(blockIdx.x) * ld;
    for (int c = threadIdx.x; c < cols; c += blockDim.x) row[c] *= k;
}
```

這正是 `cudaMallocPitch` / `hipMallocPitch` 幫你做的事; BLAS 介面裡無所不在的 `lda` / `ldb` 參數存在的理由之一也是它。 `row_padding` 實測 ld = 1000 vs 1024 的頻寬差。

### Embedding lookup: 用排序換 locality

Embedding lookup 是典型的大規模 gather, index 通常不連續。 若同一 batch 內 index 可排序或按 page 分桶, 重複與相鄰的 row 會被連續處理, L2 hit rate 大幅提高; 輸出若要按原順序, 再用 inverse permutation 還原。 這是用一點額外 bookkeeping 換 coalescing/locality:

```cpp
// One warp per lookup: lane L copies elements L, L+32 of the row.
// out_pos maps the (possibly sorted) processing order back to the
// original batch order - the inverse permutation.
__global__ void embeddingGather(const float* table, const int* idx,
                                const int* out_pos, float* out, int batch) {
    int warp = (blockIdx.x * blockDim.x + threadIdx.x) / 32;
    int lane = threadIdx.x % 32;
    if (warp >= batch) return;
    const float* src = table + static_cast<size_t>(idx[warp]) * kDim;
    float* dst = out + static_cast<size_t>(out_pos[warp]) * kDim;
    for (int d = lane; d < kDim; d += 32) dst[d] = src[d];
}
```

`embedding_lookup` 用「90% 的 index 集中在 4096 個 hot rows」的偏斜分佈模擬真實 batch, 比較隨機順序與排序 + inverse permutation 兩種跑法, 並驗證輸出完全一致。 注意 trade-off: 排序讓讀取側獲得 locality, 但寫入側從連續變成 permuted — 值得與否取決於 table 有多大、分佈有多偏斜, 要實測。

<Solution>
六個 benchmark 的預期行為:

- `aosoa_layout`: SoA ≈ AoSoA ≫ AoS (stride-3 讓 AoS 掉約三倍) — 佈局決定天花板。
- `vectorized_tail`: float4 版正確處理 n = 2²⁴ + 3 (`correctness OK`), 頻寬相當或略優於 scalar, 指令數少約 4 倍 — 在更複雜的 kernel 裡這個差距會轉成效能。
